05.01
Node Roles & Cluster Architecture

Every Elasticsearch node has one or more roles that define what it does in the cluster. In production, large clusters dedicate nodes to specific roles. Click any role to see configuration details and when to use it.

node.roles: [master]
MASTER
Manages cluster state — indices, shards, nodes. Does not store data or handle queries.
node.roles: [data]
DATA
Stores shards, handles CRUD, search, and aggregations. Primary workhorse of the cluster.
node.roles: [ingest]
INGEST
Executes ingest pipelines before documents are indexed. Can be separated to isolate CPU-heavy parsing.
node.roles: []
COORDINATING
Routes requests and merges results. No data stored, no master duties. Pure traffic management.
node.roles: [data_warm]
WARM / COLD
Tiered storage roles for ILM. Warm for less-active data, cold for rarely-queried archives.
node.roles: [ml, remote_cluster_client]
ML / TRANSFORM
Dedicated nodes for machine learning jobs and data transforms. Isolates ML memory from search.
// —
EXAM: minimum master-eligible nodes — always set cluster.initial_master_nodes on first bootstrap only. For a 3-node dedicated master cluster: discovery.zen.minimum_master_nodes: 2 (quorum = N/2 + 1). Split-brain prevention requires majority.
05.02
Security — TLS, Auth & RBAC

Elasticsearch security requires TLS for transport layer (inter-node), TLS for HTTP layer (client-to-cluster), and role-based access control. The ECE exam tests enabling security, creating users/roles, and field/document-level security.

// Security Config
// elasticsearch.yml — TLS config
# Enable security (required for TLS)
xpack.security.enabled: true

# Transport layer (inter-node)
xpack.security.transport.ssl:
  enabled:           true
  verification_mode: certificate
  keystore.path:     certs/elastic-certificates.p12
  truststore.path:   certs/elastic-certificates.p12

# HTTP layer (REST API + Kibana)
xpack.security.http.ssl:
  enabled:      true
  keystore.path: certs/http.p12

---
# Generate certs with elasticsearch-certutil:
# Step 1: Create CA
./bin/elasticsearch-certutil ca
# Step 2: Create node certs from CA
./bin/elasticsearch-certutil cert --ca elastic-stack-ca.p12
# Step 3: Set built-in user passwords
./bin/elasticsearch-setup-passwords interactive
// Notes
Two TLS layers: Transport (port 9300, inter-node communication) and HTTP (port 9200, client requests). Both must be configured separately.
verification_mode: certificate (verify cert is signed by trusted CA) vs full (verify cert + hostname). Use full in production.
elasticsearch-certutil: Bundled tool for generating self-signed CA and node certificates. Creates PKCS12 (.p12) keystores. For multi-node, generate one CA and per-node certs.
Built-in users: elastic (superuser), kibana_system, logstash_system, beats_system, apm_system, remote_monitoring_user. Always set passwords on first boot.
// Role & user management
// Create a custom role
POST /_security/role/app_reader
{
  "cluster": ["monitor"],
  "indices": [
    {
      "names":      ["app-logs-*"],
      "privileges": ["read", "view_index_metadata"]
    }
  ]
}

// Create a user and assign role
POST /_security/user/alice
{
  "password": "s3cr3tP@ss",
  "roles":    ["app_reader"],
  "full_name": "Alice Smith",
  "email":    "alice@example.com"
}

// Cluster privileges (common)
// monitor     — cluster stats, health, settings
// manage      — monitor + change settings
// manage_ilm  — ILM policy CRUD
// manage_ingest — pipeline CRUD

// Index privileges (common)
// read        — search, get
// write       — index, update, delete, bulk
// create_index — create indices
// manage      — alias, mappings, settings
// all         — full control

// Check effective permissions
GET /_security/user/alice/_has_privileges
{
  "index": [{
    "names":      ["app-logs-*"],
    "privileges": ["read"]
  }]
}
// Notes
Role structure: cluster (cluster-level actions), indices (per-index privileges with wildcard patterns), applications (Kibana/app-level privileges), run_as (impersonation).
index names: Support wildcards — "logs-*" grants access to all logs-prefixed indices. Multiple entries can be specified for different privilege sets.
_has_privileges API: Check whether a user has specific permissions without trying to execute the action. Useful for application-level permission checking.
Native vs file realm: Native realm stores users in the cluster (API-managed). File realm uses users and roles files on disk (useful for bootstrapping). Both can coexist.
// Field & document level security
// Field-level security — restrict visible fields
POST /_security/role/hr_reader
{
  "indices": [{
    "names":      ["employees"],
    "privileges": ["read"],
    "field_security": {
      "grant": ["name", "department", "title"]
    }
  }]
}
// Users with this role CANNOT see salary, ssn, etc.
// grant: explicit allow-list of fields
// except: deny specific fields, allow rest

// Document-level security — restrict visible docs
POST /_security/role/eu_data_reader
{
  "indices": [{
    "names":      ["customers"],
    "privileges": ["read"],
    "query": {
      "term": { "region": "eu" }
    }
  }]
}
// Searches through eu_data_reader ONLY return
// documents where region == "eu"
// Non-EU docs are invisible — not even counted

// Combining FLS + DLS
"indices": [{
  "names":      ["customers"],
  "privileges": ["read"],
  "field_security": {
    "grant": ["name", "email", "region"]
  },
  "query": { "term": { "region": "eu" }}
}]
// Notes
Field-level security (FLS): Controls which fields a user can see in query results. grant: allowlist. except: denylist (grant everything except listed). Wildcards supported.
Document-level security (DLS): The query acts as a filter — only documents matching the query are visible to this role. Users can't see other documents even if they know the IDs.
Performance: DLS adds a filter to every query executed by the role. FLS adds field masking. Both have performance overhead — avoid on very high-throughput indices.
EXAM TRAP: DLS makes non-matching docs completely invisible — they don't appear in counts either. A user with DLS sees hits.total reflecting only matching docs, not the true index count.

Shard Allocation & Cluster Health

Shard allocation determines where primary and replica shards live across nodes. Understanding why shards become unassigned — and how to fix it — is a core ECE exam skill.

// Allocation & Health APIs
// Health & shard diagnostics
// Overall cluster health
GET /_cluster/health
// green  = all shards assigned
// yellow = all primaries assigned, some replicas not
// red    = some primary shards unassigned

// Health for specific index
GET /_cluster/health/my_index?level=shards

// List all unassigned shards
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason

// Why is a shard unassigned?
GET /_cluster/allocation/explain
{
  "index": "my_index",
  "shard": 0,
  "primary": true
}

// Node-level shard overview
GET /_cat/nodes?v&h=name,node.role,heap.percent,disk.avail,shards

// Index-level shard stats
GET /_cat/shards/my_index?v&s=state
// Health states decoded
GREEN: All primary and replica shards are assigned and active. Cluster is fully operational.
YELLOW: All primaries assigned, but one or more replica shards are unassigned. Common on single-node clusters (no node to place replicas on). Data is safe but redundancy is reduced.
RED: One or more PRIMARY shards are unassigned. Searches on affected indices will return partial or no results. Requires immediate attention.
allocation/explain: The most important diagnostic API. Tells you exactly WHY a shard can't be allocated — disk space, node attributes, replica constraints, etc. Always start here for unassigned shards.
// Allocation settings & awareness
// Disable allocation (e.g., for rolling restart)
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "none"
  }
}
// values: all (default), primaries, new_primaries, none

// Re-enable after maintenance
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "all"
  }
}

// Shard allocation awareness (rack/zone)
// In elasticsearch.yml:
node.attr.rack: rack1

// Cluster setting:
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.awareness.attributes": "rack"
  }
}
// ES spreads primaries+replicas across different racks

// Forced awareness (don't place on same rack)
"cluster.routing.allocation.awareness.force.rack.values":
  "rack1,rack2,rack3"

// Per-index allocation filtering
PUT /warm_data/_settings
{
  "index.routing.allocation.require.node_type": "warm"
}
// Notes
allocation.enable: "none": Stops ALL shard movements. Use before rolling restarts to prevent ES from frantically reassigning shards while you take nodes down one at a time.
Allocation awareness: Ensures replicas are placed on nodes with different attribute values (e.g., different racks). Provides rack-level resilience — if a rack fails, the remaining rack still has all primary or replica shards.
Forced awareness: Prevents ES from placing all copies of a shard on the same rack even if other racks are temporarily unavailable. Avoids over-allocation to one rack during maintenance.
require/include/exclude filters: require — must match. include — may be placed there. exclude — must NOT be placed there. Used for tiered storage (hot/warm/cold).
// Manual reroute & disk thresholds
// Manually move a shard
POST /_cluster/reroute
{
  "commands": [{
    "move": {
      "index":       "my_index",
      "shard":       0,
      "from_node":   "node-1",
      "to_node":     "node-2"
    }
  }]
}

// Force-assign unassigned primary (data loss risk!)
POST /_cluster/reroute
{
  "commands": [{
    "allocate_stale_primary": {
      "index":          "my_index",
      "shard":          0,
      "node":           "node-1",
      "accept_data_loss": true
    }
  }]
}

// Disk-based allocation thresholds
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.disk.threshold_enabled": true,
    "cluster.routing.allocation.disk.watermark.low":  "85%",
    "cluster.routing.allocation.disk.watermark.high": "90%",
    "cluster.routing.allocation.disk.watermark.flood_stage": "95%"
  }
}
// Notes
allocate_stale_primary: Last resort for unrecoverable shard loss. Forces allocation to a node that has an old copy of the shard. accept_data_loss: true acknowledges you may lose recent writes. Use only after all recovery options exhausted.
Disk watermarks: low (85%) = no new shards allocated here. high (90%) = existing shards moved away. flood_stage (95%) = index becomes read-only (index.blocks.read_only_allow_delete: true). Fix: add disk space or delete data, then clear the block.
Clear flood-stage block: After freeing disk space, run: PUT /index/_settings { "index.blocks.read_only_allow_delete": null } to re-enable writes.

Snapshot Repositories & SLM

Snapshots are the primary backup mechanism for Elasticsearch. Snapshot Lifecycle Management (SLM) automates snapshot creation and retention. The ECE exam tests repository registration, taking/restoring snapshots, and configuring SLM policies.

// Snapshot API
// Register snapshot repository
// Shared filesystem repository
PUT /_snapshot/my_backup
{
  "type": "fs",
  "settings": {
    "location":             "/mnt/snapshots",
    "compress":             true,
    "max_restore_bytes_per_sec": "40mb",
    "max_snapshot_bytes_per_sec": "40mb"
  }
}
// path.repo must be set in elasticsearch.yml FIRST:
// path.repo: ["/mnt/snapshots"]

// S3 repository (requires repository-s3 plugin)
PUT /_snapshot/s3_backup
{
  "type": "s3",
  "settings": {
    "bucket":   "my-es-snapshots",
    "region":   "us-east-1",
    "base_path": "prod/cluster1"
  }
}
// Also: gcs (Google), azure, hdfs

// Verify repository is working
POST /_snapshot/my_backup/_verify
// Notes
path.repo REQUIRED: For fs repositories, you must whitelist the path in elasticsearch.yml on ALL nodes before registering. Without this, registration fails with a security error.
compress: true: Compresses snapshot metadata (not data by default). Add compress: true to reduce storage for metadata-heavy snapshots.
_verify: Tests that all nodes can access the repository location. Run after registration to confirm shared storage is correctly mounted on every node.
Throttling: max_snapshot_bytes_per_sec and max_restore_bytes_per_sec prevent snapshots from saturating network/disk IO. Default: 40mb per node.
// Take & restore snapshots
// Take snapshot of all indices
PUT /_snapshot/my_backup/snapshot_1
{
  "indices":             "*",
  "ignore_unavailable":  true,
  "include_global_state": false,
  "metadata": {
    "taken_by": "admin",
    "reason":   "pre-upgrade backup"
  }
}

// Snapshot specific indices
PUT /_snapshot/my_backup/snapshot_2
{
  "indices": ["orders", "customers", "products-*"]
}

// Check snapshot status
GET /_snapshot/my_backup/snapshot_1
// state: IN_PROGRESS | SUCCESS | FAILED | PARTIAL

// Restore snapshot (index must not exist or be closed)
POST /_snapshot/my_backup/snapshot_1/_restore
{
  "indices":          "orders",
  "rename_pattern":   "(.+)",
  "rename_replacement": "restored_$1",
  "include_global_state": false,
  "index_settings": {
    "index.number_of_replicas": 0
  }
}
// rename_pattern allows restoring alongside live index
// orders → restored_orders

// Delete snapshot
DELETE /_snapshot/my_backup/snapshot_1
// Notes
include_global_state: Whether to snapshot cluster settings, index templates, ILM policies, etc. Set false for index-only backups. Set true for full cluster backup.
Restore prerequisite: The target index must NOT exist, or must be closed (POST /index/_close) before restoring. Otherwise the restore fails.
rename on restore: Use rename_pattern (regex) and rename_replacement to restore to a different name. Essential for restoring next to a live index for comparison/recovery testing.
Incremental: Snapshots are incremental — only changed segments since the last snapshot are stored. Storage usage grows slowly for stable indices.
// SLM — Snapshot Lifecycle Management
// Create SLM policy
PUT /_slm/policy/nightly_snapshots
{
  "schedule":   "0 30 1 * * ?",  // cron: 1:30 AM daily
  "name":       "<nightly-{now/d}>",  // date-named
  "repository": "my_backup",
  "config": {
    "indices":             ["*"],
    "ignore_unavailable":  true,
    "include_global_state": false
  },
  "retention": {
    "expire_after": "30d",
    "min_count":    5,
    "max_count":    50
  }
}

// Manually trigger SLM policy (test it now)
POST /_slm/policy/nightly_snapshots/_execute

// Check SLM policy status
GET /_slm/policy/nightly_snapshots

// SLM stats (snapshots taken, failed, etc.)
GET /_slm/stats

// Cron schedule format: sec min hr dom mon dow yr
// "0 30 1 * * ?" = every day at 01:30:00
// "0 0 */6 * * ?" = every 6 hours
// Notes
schedule: Quartz cron expression. 7-field format: seconds minutes hours day-of-month month day-of-week year. "?" = don't care for that field.
name template: <...> syntax with date math. <nightly-{now/d}> produces names like "nightly-2024.03.15". Ensures unique, date-searchable snapshot names.
retention: expire_after — age limit. min_count — keep at least N snapshots regardless of age. max_count — never keep more than N snapshots. All three work together.
_execute: Immediately runs the SLM policy and takes a snapshot. Essential for testing the policy without waiting for the cron schedule. Returns the snapshot name.

Cross-Cluster Replication (CCR)

CCR replicates indices from a leader cluster to one or more follower clusters in near-real-time. Used for disaster recovery, geographic data distribution, and centralised reporting.

// CCR Configuration
// On the FOLLOWER cluster
// Step 1: Configure remote cluster connection
PUT /_cluster/settings
{
  "persistent": {
    "cluster.remote.leader_cluster.seeds": [
      "leader-node1:9300"
    ]
  }
}

// Step 2: Create a follower index
PUT /follower_index/_ccr/follow
{
  "remote_cluster": "leader_cluster",
  "leader_index":   "leader_index",
  "settings": {
    "index.number_of_replicas": 1
  },
  "max_read_request_operation_count": 5120,
  "max_outstanding_read_requests":   12
}

// Check CCR stats
GET /follower_index/_ccr/stats
// Look for: operations_received, failed_read_requests

// REQUIREMENTS:
// - Platinum/Enterprise license
// - Remote cluster configured
// - Leader index must be "soft-deleted" enabled
// - Follower is READ-ONLY (no direct writes)
// Notes
Architecture: The follower cluster reads from the leader's soft-deletes transaction log. Changes on the leader are replicated in near-real-time to the follower.
Follower is READ-ONLY: You cannot index directly to a follower index. All writes must go to the leader. Followers are search replicas only.
Soft deletes required: Leader indices must have index.soft_deletes.enabled: true (default since 7.0). Older indices may need to be reindexed to enable CCR.
Use cases: DR (follower in different datacenter), local reads (reduce latency by having regional followers), reporting cluster (separate analytics workload from production).
// Auto-follow patterns
// Automatically follow new indices matching a pattern
PUT /_ccr/auto_follow/logs_pattern
{
  "remote_cluster":    "leader_cluster",
  "leader_index_patterns": ["logs-*"],
  "follow_index_pattern":  "{{leader_index}}",
  "settings": {
    "index.number_of_replicas": 0
  }
}

// Any new index on the leader matching logs-*
// will automatically get a follower index created.
// Essential for time-series data / data streams.

// Check auto-follow stats
GET /_ccr/auto_follow/logs_pattern
GET /_ccr/stats

// Delete auto-follow pattern
DELETE /_ccr/auto_follow/logs_pattern
// Note: deleting the pattern does NOT unfollow
// existing follower indices — those must be paused
// or unfollowed separately.
// Notes
Auto-follow: Critical for rolling daily/monthly indices (logs-2024.01.*, logs-2024.02.*). Without auto-follow, you'd need to manually create a follower every time a new index is created.
{{leader_index}}: Template variable in follow_index_pattern — creates the follower with the same name as the leader. Can also prefix/suffix: "follower-{{leader_index}}".
Existing indices: Auto-follow only applies to NEW indices created after the pattern is set. Existing indices matching the pattern must be followed manually.
// Pause, resume, unfollow
// Pause replication (e.g., for maintenance)
POST /follower_index/_ccr/pause_follow

// Resume replication
POST /follower_index/_ccr/resume_follow
{
  "max_read_request_operation_count": 5120
}

// Unfollow — promote follower to a regular index
// Step 1: Pause the follow
POST /follower_index/_ccr/pause_follow
// Step 2: Close the index
POST /follower_index/_close
// Step 3: Unfollow (converts to regular index)
POST /follower_index/_ccr/unfollow
// Step 4: Re-open
POST /follower_index/_open
// Now writable as a normal index

// Forget follower (on LEADER — stop tracking)
POST /leader_index/_ccr/forget_follower
{
  "follower_cluster":       "follower_cluster",
  "follower_index":         "follower_index",
  "follower_index_uuid":    "...",
  "leader_remote_cluster":  "leader_cluster"
}
// Notes
Pause vs unfollow: Pause keeps the follower relationship intact but stops replication temporarily. Unfollow severs the relationship permanently and makes the index writable.
Failover pattern: When the leader cluster fails, use the unfollow workflow on the follower to promote it to a writable index. Applications redirect writes to the former follower.
Unfollow sequence is critical: Must pause → close → unfollow → open in that exact order. Skipping close causes unfollow to fail.

01
Security · Roles · Field-Level Security
Create a role support_agent that: (1) can read and search the tickets-* indices, (2) can only see the fields ticket_id, status, subject, created_at — NOT customer_email or internal_notes, (3) can only see tickets where assigned_team: "support". Then create a user agent_bob with this role.
// Solution
// Step 1: Create role with FLS + DLS
POST /_security/role/support_agent
{
  "cluster": ["monitor"],
  "indices": [
    {
      "names":      ["tickets-*"],
      "privileges": ["read", "view_index_metadata"],
      "field_security": {
        "grant": [
          "ticket_id",
          "status",
          "subject",
          "created_at"
        ]
      },
      "query": {
        "term": {
          "assigned_team": "support"
        }
      }
    }
  ]
}

// Step 2: Create user
POST /_security/user/agent_bob
{
  "password":  "Str0ngP@ss!",
  "roles":     ["support_agent"],
  "full_name": "Bob Smith",
  "email":    "bob@support.example.com"
}

// Step 3: Verify permissions
GET /_security/user/agent_bob/_has_privileges
{
  "index": [{
    "names":      ["tickets-2024"],
    "privileges": ["read"]
  }]
}
Key decisions: Both field_security and query can coexist in the same index entry — they apply AND logic. grant lists ONLY allowed fields; customer_email and internal_notes are not listed so they're invisible. The DLS query filters documents — agent_bob literally cannot see tickets from other teams even if they know the IDs. view_index_metadata allows the user to see index metadata (needed for Kibana to function). cluster: monitor lets the user view cluster health in Kibana.
02
Snapshots · SLM · Restore
Register an S3 snapshot repository named prod_backup using bucket es-prod-snapshots. Create an SLM policy daily_backup that runs every night at 2 AM UTC, snapshots all indices, keeps snapshots for 14 days with a minimum of 3. Then restore just the orders index from snapshot_20240315, renaming it to orders_restored.
// Solution
// Step 1: Register S3 repository
PUT /_snapshot/prod_backup
{
  "type": "s3",
  "settings": {
    "bucket":   "es-prod-snapshots",
    "region":   "us-east-1",
    "base_path": "cluster/production"
  }
}

// Step 2: Verify repository
POST /_snapshot/prod_backup/_verify

// Step 3: Create SLM policy
PUT /_slm/policy/daily_backup
{
  "schedule":   "0 0 2 * * ?",
  "name":       "<daily-{now/d}>",
  "repository": "prod_backup",
  "config": {
    "indices":             ["*"],
    "ignore_unavailable":  true,
    "include_global_state": false
  },
  "retention": {
    "expire_after": "14d",
    "min_count":    3,
    "max_count":    30
  }
}

// Step 4: Restore only 'orders' index, renamed
POST /_snapshot/prod_backup/snapshot_20240315/_restore
{
  "indices":            "orders",
  "rename_pattern":     "orders",
  "rename_replacement": "orders_restored",
  "include_global_state": false,
  "index_settings": {
    "index.number_of_replicas": 0
  }
}

// Monitor restore progress
GET /_cat/recovery/orders_restored?v
Key points: S3 repo requires the repository-s3 plugin installed on all nodes. _verify confirms all nodes can reach S3. The cron "0 0 2 * * ?" = 2:00:00 AM daily (seconds minutes hours). rename_pattern is a literal string (not regex by default here) matching "orders" → renamed to "orders_restored". Set replicas to 0 for fast restore on a single-node test environment. min_count: 3 ensures we always keep at least 3 snapshots even after expire_after kicks in.
03
Shard Allocation · Rolling Restart · Health Diagnosis
Perform a rolling restart of a 3-node cluster without data loss: (1) disable shard allocation, (2) perform a synced flush, (3) restart a node, (4) wait for the node to rejoin and shards to recover, (5) re-enable allocation. Also show how to diagnose and fix an unassigned shard caused by disk watermark.
// Solution
// ─── ROLLING RESTART PROCEDURE ───

// Step 1: Disable shard allocation
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "none"
  }
}

// Step 2: Flush all indices
POST /_flush

// Step 3: Restart the node (OS-level — outside ES)
// sudo systemctl restart elasticsearch

// Step 4: Wait for node to rejoin cluster
GET /_cat/nodes?v
// Confirm the restarted node appears in the list

// Step 5: Re-enable allocation
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "all"
  }
}

// Step 6: Check recovery
GET /_cat/recovery?v&active_only=true
GET /_cluster/health?wait_for_status=green&timeout=60s

// ─── DIAGNOSE DISK WATERMARK ISSUE ───

// Step 1: Find unassigned shards
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason

// Step 2: Get allocation explanation
GET /_cluster/allocation/explain
{
  "index":   "my_index",
  "shard":   0,
  "primary": false
}
// Response will indicate: "disk usage exceeded...
// high watermark threshold [90%]"

// Step 3: Raise watermarks temporarily (or free disk)
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.high": "95%"
  }
}

// Step 4: If index was flood-staged (read-only), clear it
PUT /my_index/_settings
{
  "index.blocks.read_only_allow_delete": null
}

// Step 5: After freeing disk, restore defaults
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.high": null
  }
}
Rolling restart key points: Disable allocation BEFORE shutting down the node — without this, ES will immediately start moving shards to other nodes when the node disappears, causing unnecessary network IO. Flush reduces the recovery work when the node comes back. Re-enable allocation AFTER the node has rejoined (visible in _cat/nodes). Repeat for each node. Disk watermark fix: use transient settings for temporary adjustments (cleared on restart). Persistent settings survive restarts. null value clears a setting back to its default.

Assessment
7 questions · Cluster Management domain · ECE exam style
QUESTION 01
A cluster has 3 dedicated master-eligible nodes. What is the minimum number that must be available to elect a master and prevent split-brain?
Correct: 2 nodes. Quorum = floor(N/2) + 1. For 3 master-eligible nodes: floor(3/2) + 1 = 2. At least 2 of the 3 must be available to elect a master and form a quorum. This prevents split-brain — if the cluster splits into partitions, only the partition with 2+ master-eligible nodes can elect a master. A single node cannot elect itself without quorum, preventing two simultaneous masters.
QUESTION 02
A user with a role that has Document Level Security applied searches an index. The index contains 10,000 documents but only 300 match the DLS query. What does the user see for hits.total.value?
Correct: 300. Document Level Security makes non-matching documents completely invisible to the user — they don't appear in counts, aggregations, or search results. From the user's perspective, the index only contains the 300 matching documents. This is by design for strict data isolation (e.g., multi-tenant setups where users should not know about data they can't access).
QUESTION 03
You want to register an fs (shared filesystem) snapshot repository at path /mnt/backups. You run the PUT /_snapshot API but get a security error. What must be done first?
Correct: add path.repo to elasticsearch.yml. Elasticsearch requires fs repository paths to be explicitly whitelisted in elasticsearch.yml via path.repo before they can be registered. This prevents arbitrary paths from being used as repositories (security measure). The setting must be on ALL nodes (since any node may need to write snapshots) and requires a restart. Only after this can the PUT /_snapshot API succeed for that path.
QUESTION 04
A node's disk usage reaches 95% (flood stage). What automatic action does Elasticsearch take on indices with shards on that node?
Correct: read-only block. When disk usage reaches the flood stage threshold (default 95%), Elasticsearch automatically sets index.blocks.read_only_allow_delete: true on all indices with shards on that node. This prevents new writes while still allowing deletes (to free space). After freeing disk, you must manually clear this block: PUT /index/_settings { "index.blocks.read_only_allow_delete": null }. The block does NOT clear automatically when disk usage drops.
QUESTION 05
When performing a rolling restart, why should you disable shard allocation (cluster.routing.allocation.enable: "none") BEFORE taking a node offline?
Correct: prevents wasteful shard movement. Without disabling allocation, when node-1 goes offline, ES immediately starts moving its shards to node-2 and node-3. When node-1 comes back up, ES moves shards again to rebalance. This is two unnecessary large data transfers for each node restarted. By disabling allocation first, ES waits patiently for the node to return, then just recovers the unchanged shards locally — far more efficient.
QUESTION 06
In CCR, you need to promote a follower index to a writable index after the leader cluster fails. What is the correct sequence of operations?
Correct: pause → close → unfollow → open. The unfollow API requires the index to be closed first. Trying to unfollow an open follower index returns an error. The full sequence is: (1) POST /index/_ccr/pause_follow — stop receiving replication, (2) POST /index/_close — close the index, (3) POST /index/_ccr/unfollow — convert to regular index, (4) POST /index/_open — make it searchable and writable. After this, the index accepts writes like any normal index.
QUESTION 07
An SLM policy has expire_after: "7d", min_count: 5, and max_count: 20. After 30 days of daily snapshots (30 snapshots exist), which snapshots are retained?
Correct: the 7 most recent (within expire_after window), guaranteed minimum 5. The three retention parameters work together: expire_after deletes snapshots older than 7 days. min_count ensures at least 5 are always kept even if they're older than 7 days (protects against accidental mass deletion). max_count caps at 20 total. After 30 days with daily snapshots: snapshots older than 7 days are deleted → ~7 remain. Since 7 > min_count (5), all within the 7-day window are kept. max_count (20) is not reached so it doesn't trim further.
