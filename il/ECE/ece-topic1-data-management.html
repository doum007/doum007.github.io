01.01Index Fundamentals

An index is a logical namespace that maps to one or more primary shards (and optionally replica shards). Every document in Elasticsearch lives in an index. The ECE exam tests your ability to create and configure indices with precise settings via the REST API — the UI alone is not sufficient.

Key index settings that must be set at creation time (immutable): number_of_shards. Key settings changeable at runtime: number_of_replicas, routing allocation, refresh interval.

// API Explorer
// Request
PUT /my_index
{
  "settings": {
    "number_of_shards":   1,   // immutable after creation
    "number_of_replicas": 1,   // can change anytime
    "refresh_interval":   "1s" // -1 to disable
  },
  "mappings": {
    "properties": {
      "title":  { "type": "text",    "analyzer": "english" },
      "price":  { "type": "float"   },
      "status": { "type": "keyword" },
      "date":   { "type": "date",
                  "format": "yyyy-MM-dd||epoch_millis" }
    }
  },
  "aliases": {
    "products": {}  // creates alias at index creation
  }
}
// Key Notes
IMMUTABLE: number_of_shards cannot change after index creation. To change shard count you must reindex into a new index.
ALIASES: Define aliases at creation time or add later with the _aliases API. Aliases enable zero-downtime reindexing.
EXAM TRAP: Setting number_of_replicas: 0 is valid and used when you have only one node (replicas would be unassigned otherwise).
VERIFY: GET /my_index — returns full settings, mappings, aliases for the index.
// Request
// Change replicas (can be done at any time)
PUT /my_index/_settings
{
  "number_of_replicas": 2
}

// Disable refresh (for bulk loading performance)
PUT /my_index/_settings
{
  "refresh_interval": "-1"
}

// Re-enable refresh after bulk load
PUT /my_index/_settings
{
  "refresh_interval": "1s"
}

// Make index read-only
PUT /my_index/_settings
{
  "index.blocks.write": true
}

// Force merge (run after bulk load + read-only)
POST /my_index/_forcemerge?max_num_segments=1
// Key Notes
BULK LOAD PATTERN: Before: replicas=0, refresh=-1. After: restore replicas, restore refresh, force merge.
FORCE MERGE: Reduces segment count for read performance. Only run on indices that will no longer receive writes (ILM warm/cold phases).
blocks.write: Used in ILM warm phase to make index read-only before force merge. GET /my_index/_settings to verify.
// Request
// Add alias to existing index
POST /_aliases
{
  "actions": [
    {
      "add": {
        "index": "my_index_v2",
        "alias": "my_index"
      }
    },
    {
      "remove": {
        "index": "my_index_v1",
        "alias": "my_index"
      }
    }
  ]
}

// Write alias — only one index can be the write target
POST /_aliases
{
  "actions": [{
    "add": {
      "index":     "logs-2024-01",
      "alias":     "logs",
      "is_write_index": true
    }
  }]
}

// Filtered alias
POST /_aliases
{
  "actions": [{
    "add": {
      "index":  "logs",
      "alias":  "error_logs",
      "filter": { "term": { "level": "error" }}
    }
  }]
}
// Key Notes
ATOMIC SWAP: The _aliases API with add+remove in one request does a zero-downtime index swap — no requests are lost.
is_write_index: When an alias points to multiple indices, exactly one must be the write target. Writes to alias without is_write_index on multi-index alias = error.
FILTERED ALIAS: Applies a query filter to all searches through the alias. Users querying error_logs only see error-level documents even though the underlying index has all levels.
VERIFY: GET /my_index/_alias — see all aliases for an index. GET /_alias/my_alias — see which indices an alias points to.
// Request
// Basic reindex
POST /_reindex
{
  "source": { "index": "old_index" },
  "dest":   { "index": "new_index" }
}

// Reindex with query filter
POST /_reindex
{
  "source": {
    "index": "old_index",
    "query": {
      "range": {
        "date": { "gte": "2024-01-01" }
      }
    }
  },
  "dest": { "index": "new_index" }
}

// Reindex from remote cluster
POST /_reindex
{
  "source": {
    "remote": {
      "host":     "http://remote:9200",
      "username": "elastic",
      "password": "changeme"
    },
    "index": "source_index"
  },
  "dest": { "index": "local_index" }
}
// Key Notes
USE CASE: Reindex when you need to change the number_of_shards, change a field mapping type, or apply a new analyzer to existing data.
DOES NOT COPY SETTINGS: _reindex only copies documents. You must create the destination index with the correct settings/mappings FIRST, then reindex into it.
REMOTE REINDEX: The remote host must be whitelisted in elasticsearch.yml via reindex.remote.whitelist.
ASYNC: Add ?wait_for_completion=false to run as background task. Use GET /_tasks/<task_id> to monitor.
01.02Mappings & Field Types

Mappings define the schema of an index — how fields are stored and indexed. The ECE exam heavily tests your ability to choose the right field type and configure advanced mapping features including multi-fields, nested objects, and dynamic templates.

TypeUse CaseAnalyzed?Aggregatable?Sortable?
textFull-text search (blog posts, descriptions)YESNO*NO*
keywordExact match, aggregations, sorting (status, ID, tag)NOYESYES
integer / long / float / doubleNumeric values, range queriesNOYESYES
dateTimestamps, date ranges (stored as epoch ms internally)NOYESYES
booleantrue/false flagsNOYESYES
objectNested JSON object (flattened — loses array relationships)FIELDSFIELDSFIELDS

// Mapping Patterns
// Multi-field mapping
PUT /products
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "english",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          },
          "raw": {
            "type": "text",
            "analyzer": "standard"
          }
        }
      }
    }
  }
}

// Now you can:
// Full-text search: title (english analyzer)
// Exact match / aggregation: title.keyword
// Standard analysis: title.raw
// Key Notes
MULTI-FIELDS: One document field stored multiple ways — full-text search AND exact-match keyword in the same field.
ignore_above: 256 — keyword fields longer than 256 chars are not indexed (prevents mapping explosion from very long strings).
EXAM SCENARIO: "Users need to search products by title AND sort/aggregate by title" → text field with a .keyword multi-field.
// Nested type for array of objects
// PROBLEM: object type flattens arrays
// Doc: { "comments": [
//   {"user":"alice","likes":5},
//   {"user":"bob",  "likes":3}
// ]}
// object type stores as:
// comments.user: [alice, bob]
// comments.likes: [5, 3]
// → alice and likes:3 can incorrectly match!

// SOLUTION: nested type
PUT /blog
{
  "mappings": {
    "properties": {
      "comments": {
        "type": "nested",
        "properties": {
          "user":  { "type": "keyword" },
          "likes": { "type": "integer" }
        }
      }
    }
  }
}

// Query with nested query
GET /blog/_search
{
  "query": {
    "nested": {
      "path": "comments",
      "query": {
        "bool": {
          "must": [
            { "term":  { "comments.user": "alice" }},
            { "range": { "comments.likes": { "gt": 4 }}}
          ]
        }
      }
    }
  }
}
// Key Notes
NESTED vs OBJECT: Use nested when you need to query across fields of the same array element. Object type loses per-element relationships.
COST: Nested documents are stored as hidden separate documents. Each nested object = one extra Lucene doc. High cardinality nested arrays = performance hit.
EXAM SCENARIO: "Configure index so it properly maintains relationships of nested arrays of objects" → type: nested + nested query.
ALWAYS: Queries on nested fields MUST use the nested query wrapper with the correct path.
// Dynamic templates
PUT /my_index
{
  "mappings": {
    "dynamic_templates": [
      {
        "strings_as_keywords": {
          "match_mapping_type": "string",
          "mapping": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      },
      {
        "longs_as_integers": {
          "match_mapping_type": "long",
          "mapping": { "type": "integer" }
        }
      },
      {
        "named_fields": {
          "match":   "*_text",
          "mapping": {
            "type":     "text",
            "analyzer": "english"
          }
        }
      }
    ]
  }
}
// Key Notes
MATCH CONDITIONS: match_mapping_type (by inferred type), match (field name glob), path_match (dot-notated path), unmatch (exclusion glob).
COMMON EXAM TASK: "Convert all string fields to keyword type" → dynamic template with match_mapping_type: "string".
ORDERING: Dynamic templates are evaluated in ORDER. First match wins. Put most specific templates first.
vs DYNAMIC MAPPING: dynamic: true = all fields auto-mapped by ES. dynamic: "strict" = unmapped fields cause an error. dynamic: false = unmapped fields ignored (not indexed but still in _source).
// Runtime fields
// Runtime field — computed at query time, not indexed
PUT /logs
{
  "mappings": {
    "runtime": {
      "day_of_week": {
        "type": "keyword",
        "script": {
          "source": """
            emit(doc['@timestamp'].value
              .dayOfWeekEnum
              .getDisplayName(
                TextStyle.FULL,
                Locale.ROOT
              )
            );
          """
        }
      }
    },
    "properties": {
      "@timestamp": { "type": "date" },
      "message":    { "type": "text"  }
    }
  }
}

// OR define at query time (no schema change)
GET /logs/_search
{
  "runtime_mappings": {
    "response_code": {
      "type": "long",
      "script": "emit(Integer.parseInt(doc['raw_code'].value))"
    }
  },
  "query": { "range": { "response_code": { "gte": 500 }}}
}
// Key Notes
RUNTIME FIELDS: Computed at query time using Painless. Not stored in the index. Useful for schema evolution without reindexing.
PERFORMANCE: Runtime fields scan all matching docs and compute on-the-fly. Much slower than indexed fields for large datasets. Use indexed fields for frequent queries.
TWO WAYS: (1) Define in mapping — available always. (2) Define in query body under runtime_mappings — ad-hoc, query-scoped only.
EXAM TIP: "Write a search that utilizes a runtime field" — define runtime_mappings in the search body and then reference the runtime field in query/aggs/sort.

MAPPING EXPLOSION WARNING: Never allow user-controlled field names to be dynamically mapped — this can create thousands of mappings and degrade performance. Use dynamic: false or dynamic: "strict" on objects with user-controlled content.

01.03Index Templates & Component Templates

Index templates automatically apply settings, mappings, and aliases to new indices matching an index pattern. The modern approach uses composable templates (index templates + component templates), which replaced legacy _template in v7.8+. The ECE exam uses the modern composable template API.

// Template API
// Step 1: Create reusable component templates
// Component template for common settings
PUT /_component_template/common_settings
{
  "template": {
    "settings": {
      "number_of_shards":   1,
      "number_of_replicas": 1
    }
  }
}

// Component template for common mappings
PUT /_component_template/common_mappings
{
  "template": {
    "mappings": {
      "dynamic_templates": [{
        "strings_as_keywords": {
          "match_mapping_type": "string",
          "mapping": { "type": "keyword" }
        }
      }],
      "properties": {
        "@timestamp": { "type": "date" }
      }
    }
  }
}
// Key Notes
COMPONENT TEMPLATES: Reusable building blocks (settings, mappings, or aliases separately). Referenced by index templates via composed_of.
ADVANTAGE: Define common config once, reuse across many index templates. Change component template = all index templates using it inherit the change.
NOT AUTO-APPLIED: Component templates alone do nothing. They must be composed into an index template (PUT /_index_template) to take effect.
// Step 2: Create index template
PUT /_index_template/logs_template
{
  "index_patterns": ["logs-*"],
  "priority": 200,   // higher = wins over conflicting templates
  "composed_of": [
    "common_settings",
    "common_mappings"
  ],
  "template": {           // template-specific overrides
    "settings": {
      "index.lifecycle.name": "logs_policy"
    },
    "mappings": {
      "properties": {
        "log_level": { "type": "keyword" },
        "service":   { "type": "keyword" }
      }
    },
    "aliases": {
      "all_logs": {}
    }
  }
}

// Verify template matches an index pattern:
POST /_index_template/_simulate_index/logs-2024-01
// Key Notes
PRIORITY: When multiple templates match an index name, the one with the highest priority wins. Built-in templates (metrics-*, logs-*) have priority 100 — use 200+ for custom templates to override them.
_simulate_index: Dry-run what settings/mappings would be applied to a hypothetical index. Use this to verify your template before creating real indices.
LEGACY vs MODERN: PUT /_template (legacy, use for v7 compat only). PUT /_index_template (modern composable). ECE exam uses modern API.
index.lifecycle.name: Link ILM policy to template so all new matching indices automatically follow the lifecycle.
// Data stream + template
// Index template must have data_stream: {}
PUT /_index_template/metrics_template
{
  "index_patterns": ["metrics-*"],
  "data_stream": {},   // enables data stream mode
  "priority": 200,
  "template": {
    "settings": {
      "number_of_shards":   1,
      "number_of_replicas": 1
    },
    "mappings": {
      "properties": {
        "@timestamp": { "type": "date" }
        // @timestamp is REQUIRED for data streams
      }
    }
  }
}

// Create data stream (just index a doc OR)
PUT /_data_stream/metrics-server-prod

// Index a doc — always use POST, not PUT
POST /metrics-server-prod/_doc
{
  "@timestamp": "2024-03-15T10:00:00Z",
  "cpu_pct":    42.5
}

// Rollover current write index
POST /metrics-server-prod/_rollover
// Key Notes
DATA STREAMS: Managed time-series indices that automatically roll over. Name pattern: .ds-<name>-<date>-NNN (backing indices). The data stream name is the alias.
REQUIRES @timestamp: Data streams MUST have an @timestamp field of type date. Documents without @timestamp will be rejected.
WRITE vs READ: Data streams route writes to the current write index (latest). Reads span ALL backing indices automatically.
NO CUSTOM ID: Data streams only support append. Cannot PUT /stream/_doc/custom_id. Cannot update or delete individual documents by ID without first converting to regular index.
ROLLOVER: Triggered automatically by ILM (max_age/max_size/max_docs) or manually via _rollover API.

ILM automates the movement of indices through lifecycle phases. The exam requires you to create complete ILM policies from scratch via the API, including all phase transitions, rollover conditions, and actions.

HOT Phase — Typical Actions & Rollover Conditions
"hot": {
  "actions": {
    "rollover": {
      "max_primary_shard_size": "50gb", // roll when shard hits 50GB
      "max_age":               "30d",   // OR index is 30 days old
      "max_docs":              1000000  // OR 1M documents
    },
    "set_priority": { "priority": 100 } // shard recovery priority
  }
}

Key: Rollover conditions are OR — first condition met triggers rollover.
Rollover REQUIRES: index must have an alias with is_write_index: true OR be a data stream.
WARM Phase — Read-only optimisation
"warm": {
  "min_age": "7d",   // wait 7 days after rollover before entering warm
  "actions": {
    "forcemerge": { "max_num_segments": 1 }, // merge into 1 segment
    "shrink":     { "number_of_shards": 1  }, // reduce shard count
    "readonly":   {},                           // set index.blocks.write: true
    "allocate": {                               // move to warm nodes
      "require": { "data": "warm" }
    },
    "set_priority": { "priority": 50 }
  }
}

min_age is measured from ROLLOVER time (not index creation), in hot-warm-cold-delete.
COLD Phase — Searchable snapshots
"cold": {
  "min_age": "30d",
  "actions": {
    "searchable_snapshot": {
      "snapshot_repository": "my_snapshot_repo"
    },
    "set_priority": { "priority": 0 }
  }
}

searchable_snapshot in cold: index is converted to a fully-cached snapshot.
Data is read from the snapshot repository (S3, GCS, Azure) but cached locally.
Reduces hot storage usage while remaining fully searchable.
FROZEN Phase — Minimal footprint
"frozen": {
  "min_age": "60d",
  "actions": {
    "searchable_snapshot": {
      "snapshot_repository": "my_snapshot_repo"
    }
  }
}

Frozen: snapshot mounted with minimal local caching. Queries are slower (fetches from
repo on demand) but uses near-zero local storage. Good for compliance/archive.
DELETE Phase — End of retention
"delete": {
  "min_age": "180d",
  "actions": {
    "delete": {
      "delete_searchable_snapshot": true // also remove the snapshot
    }
  }
}

COMPLETE POLICY: PUT /_ilm/policy/logs_policy { "policy": { "phases": { "hot": {...}, "warm": {...}, "cold": {...}, "delete": {...} } } }
// Complete ILM Policy Example
// Full policy (exam-ready)
PUT /_ilm/policy/logs_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_primary_shard_size": "10gb",
            "max_age": "1d"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "forcemerge": { "max_num_segments": 1 },
          "readonly": {},
          "set_priority": { "priority": 50 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "searchable_snapshot": {
            "snapshot_repository": "my_repo"
          }
        }
      },
      "delete": {
        "min_age": "180d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
// Key Notes
APPLY TO TEMPLATE: After creating the policy, reference it in an index template: "index.lifecycle.name": "logs_policy" in settings. Also set "index.lifecycle.rollover_alias" for non-data-stream indices.
ROLLOVER ALIAS: For regular (non-data-stream) indices, rollover requires an alias. Set index.lifecycle.rollover_alias in settings. The alias must have is_write_index: true on the current index.
CHECK STATUS: GET /logs-000001/_ilm/explain — shows current phase, step, and any errors for an index in ILM.
FORCE PHASE: POST /logs-000001/_ilm/move to manually advance an index to a specific step (useful for testing).
01.05Custom Analyzers

An analyzer is a chain of three components: Character Filters → Tokenizer → Token Filters. Understanding how text is processed is critical for building effective search. The exam requires defining and applying custom analyzers to satisfy specific requirements.

Analyzer Components
Character Filters (optional, multiple)
→ html_strip, mapping, pattern_replace
→ Applied to raw text BEFORE tokenization

Tokenizer (exactly one required)
→ standard, whitespace, pattern, keyword
→ Splits text into tokens

Token Filters (optional, multiple)
→ lowercase, stop, stemmer, synonym
→ Applied to tokens AFTER tokenization
Built-in Analyzers
standard — lowercase + stop words
english — stemming, stop words
keyword — no tokenization (whole string)
whitespace — splits on whitespace only
simple — splits on non-letters, lowercases
pattern — regex-based tokenization
language — per-language (english, french…)

Test with: GET /index/_analyze
// Custom Analyzer
// Custom analyzer in index settings
PUT /my_index
{
  "settings": {
    "analysis": {
      "char_filter": {
        "html_clean": {
          "type": "html_strip"
        }
      },
      "tokenizer": {
        "my_tokenizer": {
          "type":    "pattern",
          "pattern": "[\\W]"    // split on non-word chars
        }
      },
      "filter": {
        "en_stop": {
          "type":     "stop",
          "stopwords": "_english_"
        },
        "en_stem": {
          "type":     "stemmer",
          "language": "english"
        }
      },
      "analyzer": {
        "my_custom": {
          "type":        "custom",
          "char_filter": ["html_clean"],
          "tokenizer":   "my_tokenizer",
          "filter":      ["lowercase", "en_stop", "en_stem"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "content": {
        "type":     "text",
        "analyzer": "my_custom"
      }
    }
  }
}
// Key Notes
ORDER MATTERS: char_filter → tokenizer → filter (in array order). Filters are applied in the order listed in the filter array.
CANNOT CHANGE ANALYZER: Once an index has data, you cannot update its analyzer. Must reindex into a new index with the updated analyzer.
search_analyzer: By default the same analyzer is used for indexing and searching. Set search_analyzer on a field to use different analysis at search time (useful with synonyms).
EXAM PATTERN: "Define a custom analyzer that: strips HTML, lowercases, removes stop words, and stems" → define each component, wire them in order in the analyzer definition.
// Test analysis pipeline
// Test a named analyzer on the index
GET /my_index/_analyze
{
  "analyzer": "my_custom",
  "text": "<p>Running quickly through the fields</p>"
}

// Response shows each token after analysis:
// "tokens": [
//   {"token": "run",   "position": 0},
//   {"token": "quick", "position": 1},
//   {"token": "field", "position": 4}
// ]
// (HTML stripped, lowercased, stop words removed, stemmed)

// Test a built-in analyzer (no index needed)
GET /_analyze
{
  "analyzer": "english",
  "text": "The quick brown foxes"
}

// Test step by step (tokenizer only)
GET /_analyze
{
  "tokenizer": "standard",
  "filter":    ["lowercase", "stop"],
  "text":      "Hello World"
}
// Key Notes
_analyze API: Essential for debugging analyzer behavior. Always use this to verify your custom analyzer produces the expected tokens before applying to production data.
SEARCH CONSISTENCY: Search terms are analyzed using the same analyzer as indexing (by default). This is why "Running" matches "run" — both become "run" after analysis.
MISMATCH: If you search a text field using a term query (not match), the search term is NOT analyzed. "Running" will NOT match "run" with a term query.
// Synonym filter
PUT /products
{
  "settings": {
    "analysis": {
      "filter": {
        "product_synonyms": {
          "type":     "synonym",
          "synonyms": [
            "laptop, notebook, computer",
            "tv, television",
            "mobile => phone, smartphone"
          ]
        }
      },
      "analyzer": {
        "synonym_analyzer": {
          "type":      "custom",
          "tokenizer": "standard",
          "filter":    ["lowercase", "product_synonyms"]
        }
      }
    }
  }
}

// Equivalent: a => b means a is replaced by b
// "laptop, notebook" means each expands to the other
// BEST PRACTICE: apply synonyms at SEARCH time only
// → set as search_analyzer, not analyzer
// (avoids index bloat from synonym expansion at index time)
// Key Notes
TWO SYNONYM FORMATS:
Expansion: "a, b, c" — all expand to all
Replacement: "a => b" — a maps to b only
INDEX vs SEARCH: Applying synonyms at index time = bigger index. Applying at search time only = smaller index, but you must use search_analyzer. Best practice: synonyms at search time.
synonyms_path: Can reference a file instead of inline: "synonyms_path": "analysis/synonyms.txt" — path is relative to config directory.

{

Mapping · Nested · Reindex
The ecommerce index has a products object field (array) but queries are returning incorrect cross-object matches. Fix this by creating a new index ecommerce_fixed that correctly maintains nested relationships, then reindex the data and swap the alias.
// Solution
// Step 1: Create new index with nested type
PUT /ecommerce_fixed
{
  "settings": { "number_of_shards": 1, "number_of_replicas": 1 },
  "mappings": {
    "properties": {
      "order_id":    { "type": "keyword" },
      "customer":    { "type": "keyword" },
      "products": {
        "type": "nested",
        "properties": {
          "product_id": { "type": "keyword" },
          "name":       { "type": "text",    "fields": { "keyword": { "type": "keyword" }}},
          "price":      { "type": "float"   },
          "quantity":   { "type": "integer" }
        }
      }
    }
  }
}

// Step 2: Reindex data
POST /_reindex
{
  "source": { "index": "ecommerce"       },
  "dest":   { "index": "ecommerce_fixed" }
}

// Step 3: Atomic alias swap (zero-downtime)
POST /_aliases
{
  "actions": [
    { "add":    { "index": "ecommerce_fixed", "alias": "ecommerce" }},
    { "remove": { "index": "ecommerce",       "alias": "ecommerce" }}
  ]
}

// Step 4: Verify correct nested behaviour
GET /ecommerce_fixed/_search
{
  "query": {
    "nested": {
      "path": "products",
      "query": {
        "bool": {
          "must": [
            { "match": { "products.name":  "laptop"  }},
            { "range": { "products.price": { "lte": 500 }}}
          ]
        }
      }
    }
  }
}
Why this works: nested type stores each array element as a hidden Lucene doc, preserving field relationships. The alias swap is atomic — no downtime, no lost writes. The nested query correctly enforces that the product must be BOTH named "laptop" AND priced ≤500.
03
Custom Analyzer · Multi-field
Create an index articles with a custom analyzer that: strips HTML, lowercases, applies English stop words and stemming. The title field should use this analyzer for full-text search but also support exact-match aggregations via a keyword sub-field.
// Solution
PUT /articles
{
  "settings": {
    "analysis": {
      "char_filter": {
        "strip_html": { "type": "html_strip" }
      },
      "filter": {
        "english_stop": { "type": "stop",    "stopwords": "_english_" },
        "english_stem": { "type": "stemmer", "language": "english"   }
      },
      "analyzer": {
        "html_english": {
          "type":        "custom",
          "char_filter": ["strip_html"],
          "tokenizer":   "standard",
          "filter":      ["lowercase", "english_stop", "english_stem"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type":     "text",
        "analyzer": "html_english",
        "fields": {
          "keyword": {
            "type":         "keyword",
            "ignore_above": 256
          }
        }
      },
      "body":  { "type": "text", "analyzer": "html_english" },
      "author":{ "type": "keyword" }
    }
  }
}

// Test the analyzer
GET /articles/_analyze
{
  "analyzer": "html_english",
  "text": "<h1>The Quick Brown Foxes running</h1>"
}
// Expected tokens: ["quick", "brown", "fox", "run"]
Key points: All three components defined in settings.analysis — then referenced by name in the analyzer's char_filter/tokenizer/filter arrays. The title.keyword multi-field is not analyzed (type: keyword) so it stores the raw string for aggregations and exact-match sorting.

7 exam-style questions · Data Management domain
QUESTION 01
You need to change the number_of_shards for an existing index from 3 to 1. What is the correct approach?
Correct: Create + reindex + alias swap. number_of_shards is an immutable setting — it cannot be changed on an existing index by any method including closing/reopening. The only way is to create a new index with the desired shard count, reindex all documents into it, then atomically swap the alias to point to the new index. The shrink API can reduce shards but requires specific preconditions and only works in certain ratios.
QUESTION 02
An index stores an array of order objects, each with a product_name and price field. A query for "laptop with price < 500" is incorrectly matching orders where the laptop and the cheap item are different products. What is the root cause?
Correct: object type flattens array relationships. When type: object is used for an array, Elasticsearch flattens all values into a single array per field: product_name becomes ["laptop","shirt"] and price becomes [499,29]. A query can then match laptop (from element 1) with price 29 (from element 2). The solution is type: nested — each array element is stored as a separate hidden document, preserving the relationship between its own fields.
QUESTION 03
You are creating a dynamic template to automatically map any field whose name ends in _count as integer. Which match condition should you use?
"dynamic_templates": [{
  "counts": {
    "______": "______",
    "mapping": { "type": "integer" }
  }
}]
Correct: match: "*_count" — The match condition uses glob-style patterns against the field name. *_count matches any field name ending with _count. path_match matches dot-notated paths (e.g., "user.address.*"). match_mapping_type matches by inferred type (string, long, etc.), not name. Option D uses match_pattern: "regex" which is valid syntax but the simpler glob match is the correct choice for this name pattern.
QUESTION 04
Which of the following statements about data streams is correct?
Correct: data streams require a matching index template with data_stream: {}. Without a template containing the data_stream: {} object, creating a data stream will fail. @timestamp is REQUIRED (not optional). Custom document IDs are not supported in data streams — only POST (auto-generated IDs) works. Writes go only to the current write index (the most recent backing index), not all backing indices.
QUESTION 05
You want multiple ILM policies to apply different lifecycle rules for different teams, but all indices should share the same base mapping. What is the most efficient approach?
Correct: shared component template + separate index templates. Component templates are designed exactly for this — define common mappings once in a component template, then reference it via composed_of in multiple index templates. Each team's index template adds its own ILM settings component. Changing the shared mapping component template propagates to all composed index templates. This avoids duplication and is the idiomatic composable template pattern.
QUESTION 06
An ILM warm phase is configured with min_age: "7d". When does this 7-day clock start?
Correct: 7 days after rollover. In a hot-warm-cold-delete policy with rollover, min_age is measured from the rollover time. The hot phase contains the rollover action; once the index rolls over it becomes non-write. The warm phase min_age of "7d" means: enter warm 7 days after rollover occurred. This is distinct from index creation time. For indices without rollover (no hot phase), min_age is measured from index creation.
QUESTION 07
You define a runtime field hour_of_day in a search request's runtime_mappings. How does this differ from a runtime field defined in the index mappings?
Correct: query-time runtime fields only exist for that request. Defining runtime_mappings in a search body creates an ephemeral field that exists only during query execution — it does not modify the index mapping. Mapping-level runtime fields persist in the schema and are available to all queries without re-definition. Both are computed at query time (neither is pre-computed or stored). Both can be used in queries, aggregations, and sorting.
