Ingest Pipelines — Architecture

An ingest pipeline is a series of processors that transform documents before they are indexed. Pipelines run on ingest nodes (any node with the ingest role) and are defined at the cluster level. Every document passes through each processor in order — if any processor fails, the document is rejected unless ignore_failure: true is set.

// Pipeline attachment methods
// Method 1: Per-request pipeline parameter
POST /logs/_doc?pipeline=my_pipeline
{ "message": "192.168.1.1 GET /index.html 200" }

// Method 2: Bulk API with pipeline param
POST /_bulk?pipeline=my_pipeline
{ "index": { "_index": "logs" } }
{ "message": "..." }

// Method 3: Default pipeline on index
PUT /logs/_settings
{
  "index.default_pipeline": "my_pipeline"
}

// Method 4: Final pipeline (always runs last)
PUT /logs/_settings
{
  "index.final_pipeline": "cleanup_pipeline"
}
default_pipeline: Runs on every document indexed to this index unless the request specifies a different pipeline or pipeline: _none.
final_pipeline: Always runs AFTER the default or requested pipeline. Cannot be overridden by the indexing request. Use for mandatory post-processing.
pipeline: "_none": Explicitly disables pipeline processing for a request, bypassing even the default_pipeline.
// grok vs dissect
// grok — regex-based named capture (slower, flexible)
{
  "grok": {
    "field":   "message",
    "patterns": [
      "%{IP:client_ip} %{WORD:method} %{URIPATH:path} %{NUMBER:status:int}"
    ]
  }
}
// Built-in patterns: IP, NUMBER, WORD, URIPATH, etc.
// :type conversion: %{NUMBER:status:int}

// dissect — delimiter-based (faster, structured logs)
{
  "dissect": {
    "field":   "message",
    "pattern": "%{client_ip} %{method} %{path} %{status}"
  }
}
// Uses literal delimiters — much faster than grok
// Use dissect when log format is predictable/fixed
grok: Based on Oniguruma regex with named patterns. Supports built-in pattern library (%{IP}, %{NUMBER}, etc). Slower but handles variable-format logs.
dissect: Token-based, delimiter-driven. 10–100x faster than grok for structured logs. No regex overhead. If log format is consistent, always prefer dissect.
EXAM TIP: "Parse nginx access log" → grok with %{COMBINEDAPACHELOG}. "Parse structured CSV-like log" → dissect. Know both syntaxes.
// Enrichment processors
// geoip — add geo coords from IP address
{
  "geoip": {
    "field":        "client_ip",
    "target_field": "source.geo",
    "properties":  ["country_name", "city_name", "location"]
  }
}
// location = geo_point type, enables Kibana Maps

// set — add or overwrite a field
{
  "set": {
    "field": "event.dataset",
    "value": "nginx.access"
  }
}

// convert — change field type
{
  "convert": {
    "field":       "status_code",
    "type":        "integer",
    "ignore_missing": true
  }
}

// date — parse date string to @timestamp
{
  "date": {
    "field":   "timestamp_str",
    "formats": ["dd/MMM/yyyy:HH:mm:ss Z", "ISO8601"],
    "target_field": "@timestamp"
  }
}
geoip: Enriches documents with geographic data from an IP address. Requires the GeoIP database (bundled with Elasticsearch). location produces a geo_point field used by Kibana Maps.
set + copy_from: Can copy value from another field: "copy_from": "source_field". Or set a literal value: "value": "constant".
convert type options: integer, long, float, double, string, boolean, ip, auto. Use auto to detect type. Use ignore_missing: true to skip if field absent.
date formats array: Tried in order — first match wins. Supports Joda-Time patterns and named formats (ISO8601, UNIX, UNIX_MS, TAI64N).
// Cleanup processors
// remove — delete fields
{
  "remove": {
    "field":          ["message", "temp_field"],
    "ignore_missing": true
  }
}

// rename — rename a field
{
  "rename": {
    "field":        "host",
    "target_field": "host.name"
  }
}

// trim — strip leading/trailing whitespace
{ "trim": { "field": "username" } }

// lowercase / uppercase
{ "lowercase": { "field": "method" } }

// drop — discard the entire document
{
  "drop": {
    "if": "ctx.status == 200"
  }
}
// The document is not indexed at all
// Use for filtering out noise (health checks, etc.)
remove with array: Can remove multiple fields in one processor using an array. Use ignore_missing: true if field may not exist.
drop processor: Most powerful filtering tool. Use the "if" condition to drop entire documents that match criteria (e.g., health check requests). Dropped docs are never indexed.
Conditional "if": All processors support an "if" field containing a Painless expression. The processor only runs if the condition evaluates to true. ctx = the document. Example: "if": "ctx.status != null && ctx.status == 200"
// Script processor + on_failure
// script processor — arbitrary Painless logic
{
  "script": {
    "source": """
      if (ctx.containsKey('bytes_in') &&
          ctx.containsKey('bytes_out')) {
        ctx.total_bytes = ctx.bytes_in +
                          ctx.bytes_out;
      }
    """
  }
}

// on_failure — handle processor errors
{
  "grok": {
    "field": "message",
    "patterns": ["%{IP:client_ip} ..."]
  },
  "on_failure": [
    {
      "set": {
        "field": "parse_error",
        "value": "{{_ingest.on_failure_message}}"
      }
    }
  ]
}

// Pipeline-level on_failure (fallback for all)
PUT /_ingest/pipeline/my_pipeline
{
  "processors": [ ... ],
  "on_failure": [
    { "set": { "field": "_index", "value": "failed-docs" }}
  ]
}
script processor: Full Painless access to the document via ctx. Can create new fields, modify existing ones, perform calculations. Slower than native processors — use only when no native processor fits.
on_failure (processor-level): Runs if that specific processor fails. Execution continues with the next processor after on_failure completes.
on_failure (pipeline-level): Catches unhandled failures from any processor. Common pattern: route failed docs to a different index by setting ctx._index.
{{_ingest.on_failure_message}}: Template variable inside on_failure that contains the error message from the failed processor. Use with a set processor to capture the error reason in the document.

EXAM WORKFLOW: Always simulate first. Create pipeline → simulate with test docs → verify output → apply to index via default_pipeline setting or per-request ?pipeline= param.

Painless Scripting

Painless is Elasticsearch's purpose-built scripting language — designed to be fast, safe, and easy to use. It appears in ingest pipelines, runtime fields, update requests, aggregation scripts, and search queries. The ECE exam tests Painless in multiple contexts.

// Painless Contexts
// Ingest pipeline script processor
// ctx = the document being processed
// ctx.field_name to read/write fields

// Example: compute derived field
{
  "script": {
    "source": """
      // Calculate request rate (reqs per sec)
      if (ctx.duration_ms != null && ctx.duration_ms > 0) {
        ctx.req_rate = ctx.request_count /
                       (ctx.duration_ms / 1000.0);
      }
    """
  }
}

// Set @timestamp from epoch milliseconds
{
  "script": {
    "source": """
      if (ctx.epoch_ms != null) {
        ctx['@timestamp'] = Instant
          .ofEpochMilli(ctx.epoch_ms)
          .toString();
      }
    """
  }
}

// Tag document based on field value
{
  "script": {
    "source": """
      if (ctx.status >= 500) {
        ctx.event_category = 'error';
      } else if (ctx.status >= 400) {
        ctx.event_category = 'warning';
      } else {
        ctx.event_category = 'info';
      }
    """
  }
}
// Notes
ctx object: In ingest context, ctx is a map representing the document. Read with ctx.field or ctx['field'] (use bracket notation for fields with dots or special chars).
Null checks: Always check for null before operating on optional fields. ctx.containsKey('field') or ctx.field != null.
No return: Ingest scripts modify ctx directly — they don't return values. Contrast with runtime field scripts which use emit().
Available classes: Math, String methods, Collections, Instant (for dates), ZonedDateTime, and all Java primitives.
// Update API with Painless
// ctx._source = document in update context

// Increment a counter
POST /products/_update/123
{
  "script": {
    "source": "ctx._source.view_count++"
  }
}

// Update with params (safer than inline values)
POST /products/_update/123
{
  "script": {
    "source": "ctx._source.price = params.new_price",
    "params": { "new_price": 29.99 }
  }
}

// Append to an array field
POST /posts/_update/456
{
  "script": {
    "source": """
      if (!ctx._source.tags.contains(params.tag)) {
        ctx._source.tags.add(params.tag);
      }
    """,
    "params": { "tag": "featured" }
  }
}

// Conditional: delete document if expired
POST /sessions/_update/789
{
  "script": {
    "source": """
      if (ctx._source.expires_at < params.now) {
        ctx.op = 'delete';
      }
    """,
    "params": { "now": 1709856000000 }
  }
}
// Notes
ctx._source: In update context, the document is accessed via ctx._source (not ctx directly). Read and write fields through this object.
params: Always pass dynamic values through params rather than string-interpolating them into the script source. Params are compiled once — much faster for repeated execution.
ctx.op: Control the operation outcome. Set to "delete" to delete the doc, "noop" to make no change (no write). Default is "update".
UPSERT: Use upsert alongside script to handle the case where the document doesn't exist: { "script": {...}, "upsert": { "view_count": 1 } }
// Runtime field scripts (emit)
// Runtime fields use doc[] and emit()
// doc['field'].value = typed field access

// Extract hour from timestamp
"runtime_mappings": {
  "hour_of_day": {
    "type": "long",
    "script": {
      "source": """
        emit(doc['@timestamp'].value.getHour());
      """
    }
  }
}

// Parse IP octets for subnet matching
"first_octet": {
  "type": "keyword",
  "script": {
    "source": """
      String ip = doc['source.ip'].value;
      int dot = ip.indexOf('.');
      if (dot > 0) {
        emit(ip.substring(0, dot));
      }
    """
  }
}

// Combine two fields
"full_name": {
  "type": "keyword",
  "script": """
    emit(doc['first_name'].value + ' ' +
         doc['last_name'].value);
  """
}
// Notes
emit(): The ONLY way to output a value in runtime field scripts. Must call emit() — not return(). Can call emit() multiple times for multi-value fields.
doc['field'].value: Typed access to indexed field values. Returns the correct Java type (ZonedDateTime for dates, double for numeric, String for keyword). Always prefer doc[] over params._source in runtime fields.
doc[] vs params._source: doc[] reads from the field data cache (fast). params._source reads the raw stored _source (slower, requires _source stored). Runtime fields should use doc[].
Missing fields: Check doc['field'].size() != 0 before accessing doc['field'].value to avoid NullPointerException when field is absent.
// Common Painless patterns
// Null-safe field access
if (ctx.containsKey('field') && ctx.field != null) {
  // safe to use ctx.field
}

// String operations
ctx.lower_msg = ctx.message.toLowerCase();
ctx.trimmed   = ctx.message.trim();
boolean found = ctx.message.contains("ERROR");
int pos       = ctx.message.indexOf(":");
String sub    = ctx.message.substring(0, pos);

// Numeric conversions
ctx.as_int    = Integer.parseInt(ctx.str_val);
ctx.as_double = Double.parseDouble(ctx.str_val);
ctx.formatted = String.format("%.2f", ctx.price);

// List/Array operations
ctx.tags.add("new_tag");
ctx.tags.remove("old_tag");
boolean has = ctx.tags.contains("featured");
int size = ctx.tags.size();

// Date operations (ingest ctx)
ZonedDateTime dt = ZonedDateTime.parse(ctx.date_str);
ctx.year  = dt.getYear();
ctx.month = dt.getMonthValue();
ctx.day   = dt.getDayOfMonth();

// Math operations
ctx.ratio = (double) ctx.a / ctx.b;
ctx.abs   = Math.abs(ctx.diff);
ctx.round = Math.round(ctx.decimal);
// Notes
Type safety: Painless is statically typed but uses dynamic dispatch. Integer division truncates — always cast to double for fractional results: (double) a / b.
String methods: All Java String methods available. Most commonly tested: toLowerCase(), toUpperCase(), trim(), contains(), indexOf(), substring(), split(), replace().
Collections: ctx fields are accessed as Java Maps (for objects) and Lists (for arrays). Standard Java collection methods work: .add(), .remove(), .contains(), .size(), .get(index).
PERFORMANCE: Avoid expensive operations in scripts run on every document. Prefer native processors (geoip, date, convert) over script when a native alternative exists.

Reindex API & Update By Query

The _reindex and _update_by_query APIs are bulk operations that process existing documents. Both use a query to select documents and can apply scripted transformations during processing.

// Bulk Document Operations
// Reindex patterns
// Basic reindex — copy all docs
POST /_reindex
{
  "source": { "index": "old_index" },
  "dest":   { "index": "new_index" }
}

// Reindex with query filter
POST /_reindex
{
  "source": {
    "index": "logs-2023",
    "query": {
      "range": {
        "@timestamp": {
          "gte": "2023-06-01"
        }
      }
    },
    "_source": ["@timestamp", "message", "level"]
  },
  "dest": {
    "index":    "logs-2023-h2",
    "op_type": "create"
  }
}
// op_type: "create" — fails if doc ID already exists
// (prevents accidental overwrites)

// Async reindex (for large datasets)
POST /_reindex?wait_for_completion=false
{
  "source": { "index": "large_index" },
  "dest":   { "index": "new_large_index" }
}
// Returns task ID — monitor with:
GET /_tasks/<task_id>
// Notes
Does NOT copy settings/mappings. Create destination index with correct mappings BEFORE reindexing. _reindex only copies documents.
_source field list: Specify which source fields to include. Useful for migrating only a subset of fields to the new index.
op_type: "create": Only indexes documents that don't already exist in the destination (by _id). Prevents overwriting existing data during incremental reindex.
Throttling: Add "conflicts": "proceed" to skip version conflicts. Add "requests_per_second": 500 to throttle and avoid overwhelming the cluster during large reindexes.
// Reindex with script transformation
// Reindex and transform fields during copy
POST /_reindex
{
  "source": { "index": "old_products" },
  "dest":   { "index": "new_products" },
  "script": {
    "source": """
      // Rename field
      ctx._source.product_name = ctx._source.name;
      ctx._source.remove('name');

      // Convert price from cents to dollars
      ctx._source.price_usd =
        ctx._source.price_cents / 100.0;

      // Add a new classification field
      if (ctx._source.price_usd > 1000) {
        ctx._source.tier = 'premium';
      } else {
        ctx._source.tier = 'standard';
      }

      // Route to a different index based on value
      if (ctx._source.region == 'eu') {
        ctx._index = 'products-eu';
      }
    """
  }
}

// ctx._index — change destination index per document
// ctx._id    — change document ID
// ctx.op = 'noop' — skip this document
// Notes
Script in reindex: ctx._source gives access to the document. Can rename fields, transform values, add new fields, or remove fields during the copy.
Dynamic routing: Set ctx._index in the script to route different documents to different destination indices in a single reindex operation.
Skip documents: Set ctx.op = 'noop' in the script to skip a document (it won't be indexed in the destination). Useful for conditional migration.
EXAM PATTERN: "Reindex products and rename the 'name' field to 'product_name'" → _reindex with script that copies and removes. Remember to create destination mapping first.
// Update by query
// Update matching documents in-place
POST /products/_update_by_query
{
  "script": {
    "source": """
      ctx._source.price =
        ctx._source.price * params.multiplier;
    """,
    "params": { "multiplier": 1.1 }
  },
  "query": {
    "term": { "currency": "usd" }
  }
}

// Add a new field to all matching docs
POST /users/_update_by_query
{
  "script": {
    "source": "ctx._source.migrated = true"
  },
  "query": {
    "bool": {
      "must_not": [{ "exists": { "field": "migrated" }}]
    }
  }
}

// Re-apply ingest pipeline to existing documents
POST /logs/_update_by_query?pipeline=new_pipeline
{
  "query": { "match_all": {} }
}
// Runs the pipeline on every matched document
// Updates each doc with the pipeline output
// Notes
In-place update: _update_by_query modifies documents in the same index. Unlike _reindex, documents stay in place — it's a bulk partial update.
?pipeline= parameter: Re-run an ingest pipeline against existing documents. Extremely useful when you create a new pipeline and need to enrich historical data already in the index.
Version conflicts: By default, version conflicts abort the operation. Add "conflicts": "proceed" to skip conflicting documents and continue. Check the response for "version_conflicts" count.
EXAM COMPARISON:
_update_by_query = update docs IN-PLACE
_reindex = copy docs to NEW index
Use ubq for: adding new fields, applying pipelines
Use reindex for: mapping changes, new shard count
// Delete by query
// Delete all matching documents
POST /logs/_delete_by_query
{
  "query": {
    "range": {
      "@timestamp": {
        "lt": "now-90d"
      }
    }
  }
}

// Delete with conflict handling
POST /sessions/_delete_by_query
{
  "conflicts": "proceed",
  "query": {
    "term": { "expired": true }
  }
}

// Async delete for large deletions
POST /large_index/_delete_by_query?wait_for_completion=false
{
  "query": { "match_all": {} }
}
// Monitor: GET /_tasks/<task_id>
// Cancel: POST /_tasks/<task_id>/_cancel
// Notes
_delete_by_query: Removes all documents matching the query from the index. Does NOT remove the index itself. The index and its settings/mappings remain intact.
Segments not reclaimed immediately: Deleted docs are marked as deleted but disk space isn't freed until the next segment merge. Run _forcemerge after large deletions to reclaim disk space.
Alternative to delete_by_query for all docs: DELETE /index (drops entire index) is faster if you want to remove everything. Use delete_by_query only when deleting a subset.
EXAM NOTE: All three bulk operations (_reindex, _update_by_query, _delete_by_query) support: wait_for_completion=false for async, conflicts: "proceed" to skip version conflicts, and requests_per_second for throttling.

Performance-based tasks in ECE format. Build the full API solution before checking the answer.

01
Ingest Pipeline · grok · geoip · date · simulate
Create an ingest pipeline named nginx_logs that: (1) parses nginx access log format from message field using grok, extracting client_ip, http_method, request_path, status_code (as integer), and response_bytes (as integer), (2) enriches client_ip into source.geo, (3) parses the nginx timestamp into @timestamp, (4) removes the original message field. Include error handling and simulate with a test document.
// Solution
// Step 1: Create the pipeline
PUT /_ingest/pipeline/nginx_logs
{
  "description": "Parse nginx access logs",
  "processors": [
    {
      "grok": {
        "field": "message",
        "patterns": [
          "%{IP:client_ip} - - \\[%{HTTPDATE:raw_timestamp}\\] \"%{WORD:http_method} %{URIPATH:request_path}[^\"]*\" %{NUMBER:status_code} %{NUMBER:response_bytes}"
        ],
        "on_failure": [
          {
            "set": {
              "field": "parse_failure",
              "value": "{{_ingest.on_failure_message}}"
            }
          }
        ]
      }
    },
    {
      "convert": {
        "field":          "status_code",
        "type":           "integer",
        "ignore_missing": true
      }
    },
    {
      "convert": {
        "field":          "response_bytes",
        "type":           "integer",
        "ignore_missing": true
      }
    },
    {
      "date": {
        "field":          "raw_timestamp",
        "formats":        ["dd/MMM/yyyy:HH:mm:ss Z"],
        "target_field":   "@timestamp",
        "ignore_missing": true
      }
    },
    {
      "geoip": {
        "field":          "client_ip",
        "target_field":   "source.geo",
        "ignore_missing": true
      }
    },
    {
      "remove": {
        "field":          ["message", "raw_timestamp"],
        "ignore_missing": true
      }
    }
  ]
}

// Step 2: Simulate with a test document
POST /_ingest/pipeline/nginx_logs/_simulate
{
  "docs": [
    {
      "_source": {
        "message": "203.0.113.42 - - [15/Mar/2024:10:30:00 +0000] \"GET /api/users HTTP/1.1\" 200 1542"
      }
    }
  ]
}

// Step 3: Verify verbose output shows each processor's effect
POST /_ingest/pipeline/nginx_logs/_simulate?verbose=true
{ "docs": [{ "_source": { "message": "203.0.113.42 - - [15/Mar/2024:10:30:00 +0000] \"GET /api/users HTTP/1.1\" 200 1542" }}]}

// Step 4: Apply to index
PUT /nginx-access/_settings
{
  "index.default_pipeline": "nginx_logs"
}
Key decisions: grok uses standard HTTPDATE for the timestamp, extracted as raw_timestamp, then processed by the date processor into @timestamp. convert runs AFTER grok to change string values to integers. geoip targets source.geo for ECS compliance. remove cleans up both message AND the intermediate raw_timestamp. All processors use ignore_missing: true to prevent failures on malformed docs. The on_failure block captures grok parse errors.
02
Update By Query · Painless · Pipeline
The products index was ingested without a price_tier field. Add this field to all existing documents: "budget" if price < 50, "mid" if price < 200, "premium" otherwise. Then apply the existing tagging_pipeline to all documents in logs-2024 that don't yet have a processed: true flag.
// Solution
// Part 1: Add price_tier field via update_by_query
POST /products/_update_by_query
{
  "script": {
    "source": """
      if (ctx._source.price == null) {
        return;
      }
      double price = ctx._source.price;
      if (price < 50) {
        ctx._source.price_tier = 'budget';
      } else if (price < 200) {
        ctx._source.price_tier = 'mid';
      } else {
        ctx._source.price_tier = 'premium';
      }
    """
  },
  "query": {
    "bool": {
      "must_not": [
        { "exists": { "field": "price_tier" }}
      ]
    }
  }
}

// Part 2: Re-apply pipeline to unprocessed logs
POST /logs-2024/_update_by_query?pipeline=tagging_pipeline
{
  "query": {
    "bool": {
      "must_not": [
        { "term": { "processed": true }}
      ]
    }
  }
}
Key points: The query in Part 1 uses must_not + exists to target only docs without the field — idempotent re-runs won't double-process. The null check inside the script prevents errors on docs missing the price field. Part 2 uses the ?pipeline= URL parameter with _update_by_query — this re-runs the ingest pipeline on each matched document in place, updating it with the pipeline's output.

Reindex · Script Transform · Mapping
The orders index stores amount as a string (e.g. "29.99"). Create a new index orders_v2 with amount mapped as float, then reindex from orders to orders_v2 using a script that converts the string to a float. Only migrate orders from 2024 onward. Run the operation asynchronously.
// Solution
// Step 1: Create destination with correct mapping
PUT /orders_v2
{
  "settings": {
    "number_of_shards":   1,
    "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
      "order_id":   { "type": "keyword" },
      "amount":     { "type": "float"   },
      "created_at": { "type": "date"    },
      "customer":   { "type": "keyword" }
    }
  }
}

// Step 2: Reindex asynchronously with script + filter
POST /_reindex?wait_for_completion=false
{
  "source": {
    "index": "orders",
    "query": {
      "range": {
        "created_at": { "gte": "2024-01-01" }
      }
    }
  },
  "dest": { "index": "orders_v2" },
  "script": {
    "source": """
      if (ctx._source.amount != null) {
        ctx._source.amount =
          Float.parseFloat(ctx._source.amount);
      }
    """
  }
}
// Response: { "task": "abc123:45678" }

// Step 3: Monitor progress
GET /_tasks/abc123:45678

// Step 4: After completion, swap alias
POST /_aliases
{
  "actions": [
    { "add":    { "index": "orders_v2", "alias": "orders_current" }},
    { "remove": { "index": "orders",    "alias": "orders_current" }}
  ]
}
Key steps: Create destination index with correct mapping FIRST (reindex doesn't copy mappings). Add wait_for_completion=false for async. The script uses Float.parseFloat() to convert the string amount to a float value. The source query filters to 2024+ orders only. Monitor via GET /_tasks/{id}. The alias swap at the end is best practice — applications targeting orders_current switch transparently to the new index with no downtime.

QUESTION 01
A log format is 2024-03-15 ERROR user_service Connection timeout — fields separated by spaces with predictable structure. Which processor is the most performant choice to extract these fields?
Correct: dissect. When the log format has predictable, fixed delimiters, dissect is the correct choice. It uses token-based parsing with literal delimiters (no regex) and is 10–100x faster than grok. Pattern would be: "%{date} %{time} %{level} %{service} %{message}". Grok is more flexible but slower due to regex overhead. Script processor should be last resort when no native processor fits.
QUESTION 02
A pipeline processor fails at runtime. The pipeline has no on_failure defined at either the processor or pipeline level. What happens to the document?
Correct: the document is rejected. When a processor fails and there is no on_failure handler (at processor or pipeline level), the entire indexing operation fails and the document is NOT indexed. To handle failures gracefully, add on_failure at the processor level (handles just that processor's failures) or at the pipeline level (catches any unhandled failures). Alternatively, set ignore_failure: true on a processor to skip it on failure and continue processing.
QUESTION 03
In a Painless script for an ingest pipeline, what variable gives you access to the current document's fields, and how do you access a field named @timestamp?
Correct: ctx['@timestamp']. In ingest pipeline Painless scripts, the document is accessed via ctx (not ctx._source which is the update API context). For fields with special characters like @, use bracket notation: ctx['@timestamp']. ctx.normal_field also works for standard field names. doc[] is used in runtime field scripts and search context, not ingest context.
QUESTION 04
You want to add a new region field to 500,000 existing documents in the customers index based on the country_code field. The documents stay in the same index. Which API should you use?
Correct: _update_by_query. When you need to add/modify fields on existing documents IN THE SAME INDEX, _update_by_query is the right tool. It reads each matching document, applies the script, and writes the result back. _reindex is for copying to a NEW index (typically needed for mapping changes or shard count changes). _bulk with 500k individual update requests is possible but far less efficient than _update_by_query's optimized bulk processing.
QUESTION 05
You call POST /_ingest/pipeline/my_pipeline/_simulate. What does this operation do?
Correct: dry-run against sample documents. The _simulate API accepts test documents in the request body (under "docs") and runs them through the pipeline, returning the fully transformed document output. Nothing is indexed. This is the primary tool for developing and debugging pipelines. Add ?verbose=true to see the document state after each individual processor, making it easy to identify which step causes unexpected behavior.
QUESTION 06
During a reindex operation, a document in the source index has been updated by another process, causing a version conflict. You want the reindex to skip conflicting documents and continue. What parameter do you add?
Correct: "conflicts": "proceed". Adding "conflicts": "proceed" at the top level of the _reindex request instructs the API to skip documents that cause version conflicts and continue with the remaining documents. The final response will include a "version_conflicts" count showing how many were skipped. By default ("conflicts": "abort"), any version conflict stops the entire operation. This parameter works identically in _update_by_query and _delete_by_query.
QUESTION 07
In a reindex script, you set ctx.op = 'noop' for a document. What is the effect?
Correct: the document is skipped. Setting ctx.op = 'noop' in a reindex script tells Elasticsearch to skip that document — it will not be indexed to the destination. This is the standard pattern for conditional migration: only migrate documents that meet certain criteria. In update_by_query, ctx.op = 'noop' means "make no change to this document" (saves a write operation). ctx.op = 'delete' deletes the document from the source.
