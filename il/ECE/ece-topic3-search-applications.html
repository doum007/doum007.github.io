Highlighting

Highlighting returns snippets of matched text with the search terms wrapped in HTML tags. The ECE exam tests your ability to configure the highlighter type, customize tags, control fragment size, and apply highlighting to specific fields.

// Highlighting API
// Request
GET /articles/_search
{
  "query": {
    "match": { "body": "elasticsearch search" }
  },
  "highlight": {
    "pre_tags":       ["<em>"],
    "post_tags":      ["</em>"],
    "number_of_fragments": 3,
    "fragment_size":       150,
    "fields": {
      "body": {},
      "title": {
        "number_of_fragments": 0
      }
    }
  }
}

// Response: hit._highlight contains an object
// { "body": ["...<em>elasticsearch</em> query..."],
//   "title": ["Learn <em>Elasticsearch</em>"] }
// Notes
pre_tags / post_tags: Default is <em></em>. Can use any HTML. Exam may specify custom tags like <mark> or <strong>.
number_of_fragments: How many snippets to return per field. Set to 0 to return the entire field with highlights (no fragmentation — useful for short title fields).
fragment_size: Character length of each snippet. Default: 100. Larger = more context around the match.
EXAM PATTERN: "Return the title field as a complete highlighted string" → number_of_fragments: 0 on the title field.
// Highlighter type selection
// unified (default) — uses BM25 for fragment scoring
"highlight": {
  "type": "unified",
  "fields": { "body": {} }
}

// plain — uses Lucene's standard highlighter
// accurate but slower, requires term vectors or re-analysis
"highlight": {
  "type": "plain",
  "fields": { "body": {} }
}

// fvh (fast vector highlighter) — fastest for large fields
// requires term_vector: "with_positions_offsets" in mapping
"highlight": {
  "type": "fvh",
  "fields": {
    "body": {
      "matched_fields": ["body", "body.english"]
    }
  }
}

// Mapping for fvh:
"body": {
  "type": "text",
  "term_vector": "with_positions_offsets"
}
// Comparison
unified (default): Works with all query types including phrase queries. Best general-purpose choice. Uses the Lucene Unified Highlighter. Recommended unless you have a specific reason for another type.
plain: Most accurate — understands complex queries. Slowest because it re-analyzes text at highlight time. Use for complex span queries.
fvh (Fast Vector Highlighter): Fastest for large fields because it reads pre-stored term vectors instead of re-analyzing. Requires term_vector setting in mapping BEFORE indexing. Good for multi-field highlighting.
EXAM RULE: Unless specifically asked for fvh or plain, default to unified. If fvh is required, remember to update the mapping first (requires reindex).
// Advanced options
GET /docs/_search
{
  "query": { "match": { "body": "search engine" }},
  "highlight": {
    "fields": {
      "body": {
        "highlight_query": {
          "bool": {
            "should": [
              { "match": { "body": "search" }},
              { "match": { "body": "engine" }},
              { "match": { "body": "indexing" }} // extra term
            ]
          }
        },
        "order":               "score",
        "no_match_size":       150,
        "phrase_limit":        256,
        "require_field_match": false
      }
    }
  }
}
// Notes
highlight_query: Override the query used for highlighting. Useful when your main query is different from what you want highlighted (e.g., rescoring).
order: "score" — Return highest-scoring fragments first instead of in document order. Surfaces the most relevant snippets.
no_match_size: If no highlight found, return this many characters from the start of the field. Prevents empty highlight results.
require_field_match: false — Highlight fields that weren't in the query. E.g., search on title but highlight body. Default is true (only highlight queried fields).

Sorting

Sorting controls the order of returned documents. By default, results are sorted by _score descending. The exam tests single/multi-field sorting, sort by geo distance, and sorting on nested fields.

// Sort API
// Request
GET /products/_search
{
  "sort": [
    { "price":     { "order": "asc"  }},
    { "rating":    { "order": "desc" }},
    { "_score":    { "order": "desc" }},
    { "_doc": { "order": "asc"  }}
  ],
  "query": { "match_all": {} }
}

// Sort is evaluated left to right — use later fields
// as tiebreakers when earlier fields are equal.

// missing values — control with "missing" param
{ "price": { "order": "asc", "missing": "_last"  }}
{ "price": { "order": "asc", "missing": "_first" }}
{ "price": { "order": "asc", "missing": 0       }}

// Sort mode for multi-value fields
{ "prices": { "order": "asc", "mode": "min" }}
// modes: min, max, sum, avg, median
// Notes
text fields cannot be sorted. Must use keyword sub-field. "name" (text) → "name.keyword" for sorting. Sorting a text field causes an error.
_score: Include _score explicitly in the sort array if you want relevance-based tiebreaking when other fields are equal.
_doc: Sort by index order — fastest sort, no scoring. Use with scroll/pagination for performance.
missing: Controls placement of docs where the sort field is absent. "_last" puts them at the end, "_first" at the start.
mode: When a field has multiple values per doc (array), mode controls which value to sort by. Default is min for asc, max for desc.
// Geo distance sort
GET /restaurants/_search
{
  "sort": [
    {
      "_geo_distance": {
        "location": {
          "lat": 51.5074,
          "lon": -0.1278
        },
        "order":         "asc",
        "unit":          "km",
        "distance_type": "arc",
        "mode":          "min"
      }
    }
  ],
  "query": { "match_all": {} }
}

// location field must be mapped as geo_point
// "PUT /restaurants" { "mappings": {
//   "properties": {
//     "location": { "type": "geo_point" } }}}

// Also filter by geo distance:
"filter": [{
  "geo_distance": {
    "distance": "5km",
    "location": { "lat": 51.5074, "lon": -0.1278 }
  }
}]
// Notes
REQUIRES geo_point mapping. The location field must be type: geo_point. Geo sort does not work on text/keyword fields.
unit: m (meters), km, mi, yd, ft, in, cm, mm, nm. Default: m. Use "km" for most practical scenarios.
distance_type: arc (default, accurate great-circle distance) or plane (faster, less accurate for large distances).
Combining filter + sort: Use geo_distance filter in bool.filter to limit results to a radius, then _geo_distance sort to order by proximity within that radius.
// Sorting on nested fields
// Sort by min price within nested "offers" objects
GET /products/_search
{
  "sort": [
    {
      "offers.price": {
        "order": "asc",
        "mode":  "min",
        "nested": {
          "path": "offers",
          "filter": {
            "term": {
              "offers.in_stock": true
            }
          }
        }
      }
    }
  ],
  "query": { "match_all": {} }
}

// nested.path = the nested field name
// nested.filter = only consider nested docs matching this
// Notes
nested.path is required when sorting on a field inside a nested type. Without it, ES cannot resolve the sort value correctly and will error.
nested.filter: Only consider nested docs matching this filter when computing the sort value. E.g., sort by cheapest IN-STOCK offer, ignoring out-of-stock ones.
mode: "min": When a product has multiple offers, sort by the minimum (cheapest) price among all its nested offer documents.

Pagination

Elasticsearch offers three pagination mechanisms. Choosing the right one is critical — each has different tradeoffs between simplicity, consistency, and scalability. The ECE exam tests all three.

FROM / SIZE
Classic · Simple
Consistent results✗ No
Deep pagination✗ No
Stateless✓ Yes
Max offset default10,000
SEARCH_AFTER
Cursor · Efficient
Consistent results≈ PIT
Deep pagination✓ Yes
Stateless✗ No
Max offsetUnlimited
SCROLL
Legacy · Export
Consistent results✓ Yes
Deep pagination✓ Yes
Stateless✗ No
Max offsetUnlimited
// from / size pagination
// Page 1: from=0, size=10
GET /products/_search
{
  "from": 0,
  "size": 10,
  "query": { "match_all": {} },
  "sort": [{ "price": "asc" }]
}

// Page 2: from=10, size=10
GET /products/_search
{ "from": 10, "size": 10, ... }

// Page N: from = (N-1) * size

// Max: index.max_result_window (default 10,000)
// from + size must be ≤ 10,000

// Raise limit (not recommended for deep pagination):
PUT /products/_settings
{
  "index.max_result_window": 50000
}
USE WHEN: Small to medium result sets, user-facing search with page numbers, total count needed (hits.total), first <10k results only.
DEEP PAGINATION PROBLEM: from:9990 size:10 requires Elasticsearch to fetch and rank 10,000 docs from every shard, then discard 9,990. Extremely expensive at scale. Never for deep data export.
from + size > 10,000 returns an error by default. Raising max_result_window increases memory pressure — not a real fix for deep pagination needs.
// search_after with PIT
// Step 1: Open a Point in Time (PIT) for consistency
POST /products/_pit?keep_alive=1m
// Response: { "id": "46ToAwMDaWdv..." }

// Step 2: First page (no search_after yet)
GET /_search
{
  "pit": {
    "id":         "46ToAwMDaWdv...",
    "keep_alive": "1m"
  },
  "sort": [
    { "price":  "asc"  },
    { "_shard_doc": "asc" }  // tie-breaker (unique)
  ],
  "size": 10
}
// Save last hit's sort values: e.g. [29.99, 1234]

// Step 3: Next page using search_after
GET /_search
{
  "pit": { "id": "46ToAwMDaWdv...", "keep_alive": "1m" },
  "sort": [{ "price": "asc" }, { "_shard_doc": "asc" }],
  "search_after": [29.99, 1234],
  "size": 10
}

// Step 4: Close PIT when done
DELETE /_pit
{ "id": "46ToAwMDaWdv..." }
USE WHEN: Deep pagination, infinite scroll, exporting large datasets page by page where you need consistent results.
PIT (Point in Time): Freezes a view of the index at the moment of creation. Results are consistent even if new docs are indexed or old ones deleted between pages.
TIEBREAKER REQUIRED: Sort must include a unique tiebreaker as the last field. Use _shard_doc (with PIT) or _id (without PIT). Without a unique tiebreaker, results may be inconsistent.
search_after value: Taken from the last hit's sort array in the previous response. Copy the entire sort array value from the last document.
PREFERRED over scroll for new applications. More efficient, doesn't hold open a search context on every shard.
// Scroll API (legacy)
// Step 1: Initialize scroll
POST /logs-*/_search?scroll=1m
{
  "size": 1000,
  "query": { "match_all": {} },
  "sort": ["_doc"]  // fastest sort for scroll
}
// Response: { "_scroll_id": "DXF1ZXJ5Q...", hits: [...] }

// Step 2: Fetch next batch (repeat until hits empty)
POST /_search/scroll
{
  "scroll":    "1m",
  "scroll_id": "DXF1ZXJ5Q..."
}
// Repeat until hits.hits is empty []

// Step 3: Clear scroll context
DELETE /_search/scroll
{ "scroll_id": ["DXF1ZXJ5Q..."] }
DEPRECATED for new use. Scroll API is legacy — use search_after + PIT for new applications. Scroll is still tested on ECE for knowledge of existing systems.
USE WHEN: Bulk data export, migrating all documents from an index. Good for one-pass sequential reads of all documents.
RESOURCE INTENSIVE: Holds a consistent search context open on ALL shards for the scroll duration. With thousands of concurrent scrolls, this causes significant heap memory pressure.
scroll TTL: The ?scroll=1m parameter sets how long the scroll context stays alive between requests (not total), refreshed on each call. Always DELETE when done.

Fuzzy Matching & Search Templates

Fuzzy queries handle typos and spelling variations. Search templates let you define parameterized queries that can be stored and reused — critical for applications where queries are constructed at runtime.

// Fuzzy & Templates
// Fuzzy query options
// Standalone fuzzy query
GET /products/_search
{
  "query": {
    "fuzzy": {
      "title": {
        "value":          "laptoop",
        "fuzziness":      "AUTO",
        "prefix_length":  2,
        "max_expansions": 50,
        "transpositions": true
      }
    }
  }
}

// Fuzzy in match query (most common in practice)
GET /products/_search
{
  "query": {
    "match": {
      "title": {
        "query":     "laptoop gaming",
        "fuzziness": "AUTO"
      }
    }
  }
}

// fuzziness values:
// AUTO   → 0 edits for 1-2 chars
//          1 edit  for 3-5 chars
//          2 edits for 6+ chars
// 0, 1, 2 → explicit edit distance
// Notes
Levenshtein distance: Number of single-character edits (insertions, deletions, substitutions) needed to transform one string into another. "laptoop" → "laptop" = 1 substitution.
transpositions: true — Counts adjacent character swaps as one edit (Damerau-Levenshtein). "laptpo" → "laptop" = 1 transposition. Default: true.
prefix_length: Number of characters at the start that must match exactly (not fuzzy). prefix_length: 2 means "la..." must be correct, reducing false positives. Performance boost for large indices.
max_expansions: Max number of fuzzy term variations to consider. Higher = more recall, more CPU. Default: 50.
EXAM RECOMMENDATION: Add fuzziness: "AUTO" to match queries rather than using standalone fuzzy query. Match + fuzziness handles multi-term inputs more naturally.
// Search templates
// Step 1: Store a search template
PUT /_scripts/product_search
{
  "script": {
    "lang":   "mustache",
    "source": {
      "query": {
        "bool": {
          "must": [
            {
              "match": {
                "title": "{{query_string}}"
              }
            }
          ],
          "filter": [
            {{#category}}
            { "term": { "category": "{{category}}" }},
            {{/category}}
            { "range": {
              "price": {
                "gte": "{{min_price}}",
                "lte": "{{max_price}}"
              }
            }}
          ]
        }
      },
      "size": "{{size}}",
      "from": "{{from}}"
    }
  }
}

// Step 2: Execute with parameters
GET /products/_search/template
{
  "id": "product_search",
  "params": {
    "query_string": "gaming laptop",
    "category":     "electronics",
    "min_price":    500,
    "max_price":    2000,
    "size":         10,
    "from":         0
  }
}

// Render without executing (debug)
GET /_render/template
{
  "id":     "product_search",
  "params": { "query_string": "test", ... }
}
// Notes
Mustache syntax: {{variable}} — simple substitution. {{#variable}}...{{/variable}} — conditional block (renders only if variable is truthy/non-empty). {{^variable}}...{{/variable}} — inverted condition.
lang: "mustache" — always required for search templates. Templates are stored as scripts in the cluster state.
_render/template: Returns the rendered query JSON without executing it. Essential for debugging parameter substitution before running against real data.
EXAM SCENARIO: "Create a parameterized query template for product search with optional category filter" → PUT /_scripts/name with mustache, use conditional {{#category}}block{{/category}}.
msearch/template: Run multiple templates in one request using POST /_msearch/template.
// Index aliases for search
// Filtered alias — virtual view of an index
POST /_aliases
{
  "actions": [{
    "add": {
      "index":  "logs-2024",
      "alias":  "error_logs",
      "filter": {
        "term": { "level": "ERROR" }
      }
    }
  }]
}

// Now: GET /error_logs/_search returns ONLY error docs
// even though the underlying index has all log levels.

// Multi-index alias — search across multiple indices
POST /_aliases
{
  "actions": [
    { "add": { "index": "logs-2022", "alias": "all_logs" }},
    { "add": { "index": "logs-2023", "alias": "all_logs" }},
    { "add": { "index": "logs-2024", "alias": "all_logs" }}
  ]
}
// GET /all_logs/_search queries all three indices

// Routing alias — improve query performance
POST /_aliases
{
  "actions": [{
    "add": {
      "index":          "users",
      "alias":          "users_eu",
      "filter":         { "term": { "region": "eu" }},
      "search_routing": "eu"
    }
  }]
}
// Notes
Filtered aliases: Apply a query filter that automatically restricts all searches through that alias. Users cannot see documents outside the filter — useful for multi-tenancy.
Multi-index alias: One alias pointing to multiple indices. Searches span all of them transparently. Cannot use for writes unless is_write_index is set on exactly one.
Filtered alias + write: Can index through a filtered alias, but the filter doesn't prevent indexing non-matching documents — it only filters reads. The filter is purely a query-time restriction.
search_routing: Routes queries to specific shards based on the routing value. With custom shard routing, this can dramatically reduce the shards queried.

Highlighting · Sorting · Pagination

          
Write a search on the articles index that: (a) searches for "machine learning" in body with fuzzy matching, (b) highlights matches in body (max 2 fragments of 200 chars) and returns the full title highlighted, (c) sorts by published_at descending then _score, (d) returns page 2 of results (10 per page).

        

        
›

      

      

        

          
// Solution

          
GET /articles/_search
{
  "from": 10,
  "size": 10,
  "query": {
    "match": {
      "body": {
        "query":     "machine learning",
        "fuzziness": "AUTO"
      }
    }
  },
  "highlight": {
    "fields": {
      "body": {
        "number_of_fragments": 2,
        "fragment_size":       200
      },
      "title": {
        "number_of_fragments": 0
      }
    }
  },
  "sort": [
    { "published_at": { "order": "desc" }},
    { "_score":        { "order": "desc" }}
  ]
}

          
Key decisions: from: 10 = page 2 (0-indexed). fuzziness in the match query (not standalone fuzzy). number_of_fragments: 0 on title returns the complete title with highlights. Two sort fields: primary is date, secondary is relevance as tiebreaker. Highlight fields block has per-field configuration.

GET /_search
{
  "pit": {

  "sort": [
    { "@timestamp": { "order": "desc" }},
    { "_shard_doc": { "order": "asc"  }}
  ],
  "size": 20,
  "query": { "match_all": {} }

GET /_search

  "pit": {
    "id":         "46ToAwMDaWdv...",
    "keep_alive": "2m"
  },
  "sort": [
    { "@timestamp": { "order": "desc"  }},
    { "_shard_doc": { "order": "asc"   }}
  ],
  "search_after": [1709856000000, "evt_4829"],

{ "id": "46ToAwMDaWdv..." }

Create a search template named product_search that accepts query_string (required), optional max_price, and size (default 10). Then create a filtered alias in_stock_products on the products index that only exposes documents where in_stock: true. Finally, execute the template against the alias.

// Solution

          
// Step 1: Create the search template
PUT /_scripts/product_search
{
  "script": {
    "lang":   "mustache",
    "source": {
      "query": {
        "bool": {
          "must": [
            {
              "match": {
                "title": "{{query_string}}"
              }
            }
          ],

            {{#max_price}}
            {
              "range": {
                "price": { "lte": "{{max_price}}" }
              }
            }
            {{/max_price}}
          ]
        }
      },
      "size": "{{size}}"
    }
  },
  "params": {
    "size": 10
  }
}

// Step 2: Create filtered alias
POST /_aliases
{
  "actions": [{
    "add": {
      "index":  "products",
      "alias":  "in_stock_products",
      "filter": {
        "term": { "in_stock": true }
      }
    }
  }]
}

// Step 3: Execute template against alias
POST /in_stock_products/_search/template
{
  "id": "product_search",
  "params": {
    "query_string": "laptop",
    "max_price":    1500
  }
}

// Verify template renders correctly (debug):
GET /_render/template
{
  "id":     "product_search",
  "params": { "query_string": "laptop", "max_price": 1500 }
}

POST /index/_search/template (not GET). The filtered alias transparently restricts results to in_stock: true documents — the template doesn't need to know about this. Always verify with _render/template before using in production.

        

      

    

  







  
Assessment

  
7 questions · Developing Search Applications · ECE exam style


  
  

    
QUESTION 01

    
You want the title field to be returned as a complete highlighted string (not fragmented into snippets). Which configuration achieves this?

    

      
      
      
      
    

    
Correct: number_of_fragments: 0. Setting number_of_fragments to 0 tells Elasticsearch to return the entire field content as one highlighted string instead of fragmenting it. This is the canonical pattern for short fields like titles. A large fragment_size doesn't prevent fragmentation — it just makes fragments longer. Setting fragments: 1 returns only one fragment, not the whole field.

  


  
  

    
QUESTION 02

    
A search request sorts by name (a text field) in ascending order. What happens?

    

      
      
      
      
    

    
Correct: error. Sorting on a text field causes an error in Elasticsearch. Text fields are analyzed (tokenized) and their values are not stored in a format suitable for sorting. To sort on a name field, you must use a keyword sub-field: "name.keyword". If no keyword sub-field exists, you must update the mapping and reindex.

  


  
  

    
QUESTION 03

    
Which pagination method should be used for an application that needs to export all 5 million documents from an index as efficiently as possible, with consistent results?

    

      
      
      
      
    

    
Correct: Scroll API with sort: ["_doc"]. For bulk export of all documents in a sequential, single-pass operation, the Scroll API is the appropriate choice. sort: ["_doc"] uses internal index order — fastest possible sort, no scoring overhead. search_after with PIT (C) is also valid for deep pagination but is better suited for interactive use or when you need to page back and forth. The scroll API, while deprecated for new UIs, remains the standard for bulk data export. from/size is limited to 10,000 records.

  


  
  

    
QUESTION 04

    
When using search_after with a PIT, what value should you use as the sort tiebreaker to guarantee unique, stable ordering?

    

      
      
      
      
    

    
Correct: _shard_doc. When using search_after with a PIT, _shard_doc is the recommended tiebreaker. It is unique within each shard and provides deterministic ordering. _id can be used without PIT but may not be unique across shards in all cases. _seq_no is document-level versioning, not a sort field. _index doesn't guarantee uniqueness within an index.

  


  
  

    
QUESTION 05

    
What is the effect of fuzziness: "AUTO" on a search term that is 4 characters long?

    

      
      
      
      
    

    
Correct: 1 edit. AUTO fuzziness follows these rules based on term length: 0 edits for 1-2 character terms, 1 edit for 3-5 character terms, 2 edits for terms 6+ characters. A 4-character term falls in the 3-5 range, so AUTO allows 1 Levenshtein edit (insertion, deletion, substitution, or transposition if transpositions: true).

  


  
  

    
QUESTION 06

    
You create a filtered alias eu_users on the users index with filter {"term": {"region": "eu"}}. A document with region: "us" is indexed through the eu_users alias. What happens?

    

      
      
      
      
    

    
Correct: indexed but invisible through the alias. Filtered aliases apply their filter only at query time — they do NOT filter writes. Indexing a us-region document through eu_users succeeds and the document lands in the users index. However, when you search through eu_users, the filter restricts results to region: "eu" docs, so the us document won't appear. Filtered aliases provide read-level isolation, not write protection.

  


  
  

    
QUESTION 07

    
In a Mustache search template, what syntax makes a filter block appear conditionally only when the min_price parameter is provided?

    

      
      
      
      
    

    
Correct: {{#min_price}} ... {{/min_price}}. Mustache uses the section syntax {{#variable}}...{{/variable}} for conditional rendering. The block renders if the variable is truthy (non-null, non-empty, non-false). The inverted form {{^variable}}...{{/variable}} renders when the variable is falsy. Jinja2 ({% if %}) and ternary operators are not valid Mustache syntax.
