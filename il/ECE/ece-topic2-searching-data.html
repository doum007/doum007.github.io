02.01Full-Text & Term-Level Queries

The ECE exam distinguishes sharply between full-text queries (which analyze the search string) and term-level queries (which do not). Choosing the wrong type is one of the most common exam errors.

Full-Text Queries — use on text fields
match — standard full-text on one field
match_phrase — all terms in order (slop=0 default)
match_phrase_prefix — last term is prefix
multi_match — match across multiple fields
query_string — Lucene syntax (AND/OR/wildcards)
simple_query_string — lenient version of above

→ Search term IS analyzed before matching
→ Use on fields of type: text
Term-Level Queries — use on keyword/numeric/date
term — exact single value match
terms — exact match any of a list
range — gte/gt/lte/lt on numbers or dates
exists — field has a non-null value
prefix — keyword starts with string
wildcard — ? (one char) * (any chars)
regexp — Lucene regex pattern

→ Search term is NOT analyzed
→ Use on keyword, numeric, date, ip fields
// match query
GET /products/_search
{
  "query": {
    "match": {
      "description": {
        "query":    "quick brown fox",
        "operator": "and",     // default: or
        "fuzziness": "AUTO"    // typo tolerance
      }
    }
  }
}

// operator: "or"  → any term matches (default)
// operator: "and" → all terms must match
// fuzziness: AUTO → 0 edits ≤2 chars,
//                   1 edit 3-5 chars,
//                   2 edits ≥6 chars

// Shorthand (just string, operator=or, no fuzz):
"match": { "description": "quick brown fox" }
// Notes
ANALYZED: The query string "quick brown fox" is analyzed with the same analyzer as the field before matching. With english analyzer: ["quick","brown","fox"] → ["quick","brown","fox"] (stems, stops etc. applied).
TRAP: Never use match on a keyword field. "Status: Active" won't match "active" with match — keyword fields need term queries. Use match only on text fields.
operator: "and" — all three words must appear in the field. Best for precision. operator: "or" (default) — any word matches. Best for recall.
minimum_should_match: Alternative to operator — specify minimum number or percentage of terms that must match: "minimum_should_match": "75%"
// match_phrase query
GET /logs/_search
{
  "query": {
    "match_phrase": {
      "message": {
        "query": "out of memory",
        "slop":  1     // allow 1 intervening word
      }
    }
  }
}

// slop: 0 (default) — terms must be adjacent
// slop: 1 — one word can appear between terms
// slop: 2 — two words can appear between, etc.

// match_phrase_prefix — last term is a prefix
GET /logs/_search
{
  "query": {
    "match_phrase_prefix": {
      "message": "out of mem"
    }
  }
}
// Matches: "out of memory", "out of memcached"
// Use for search-as-you-type completion
// Notes
PHRASE MATCH: All terms must appear in the exact order (with slop: 0). Terms are also analyzed first. "Out of memory" → ["out","of","memory"] — all three must appear consecutively.
EXPENSIVE: Phrase queries require position information. They are slower than simple match queries. Avoid on very large indices without filters to narrow the search first.
SLOP: Slop of 1 means "out of memory" also matches "out heap of memory" (one intervening word). Each transposition also costs 1 slop.
match_phrase_prefix: The LAST term only is treated as a prefix. "quick br" matches "quick brown" and "quick bright" but NOT "qui brown".
// multi_match query
GET /articles/_search
{
  "query": {
    "multi_match": {
      "query":  "elasticsearch tutorial",
      "fields": ["title^3", "body", "tags"],
      "type":   "best_fields"
    }
  }
}

// ^3 = boost title matches 3x in scoring

// TYPES:
// best_fields   — highest single-field score wins
//                (default, good for: find best matching field)
// most_fields   — sum scores from all matching fields
//                (good for: multiple analyzed versions)
// cross_fields  — treat all fields as one big field
//                (good for: first/last name split across fields)
// phrase        — like match_phrase across fields
// phrase_prefix — like match_phrase_prefix across fields

// tie_breaker: 0.3 — partial credit for other fields
"tie_breaker": 0.3  // added to best_fields
// Notes
best_fields (default): Uses the score of the best-matching field as the score. tie_breaker adds a fraction of other fields' scores. Good for: "find docs where any field matches best".
cross_fields: Treats all fields as one document. Requires all terms to be present across fields combined. Good for: searching "John Smith" across first_name + last_name fields where each field holds one part.
BOOSTING: ^3 on title means title matches score 3x higher than body matches. Documents where the title matches the query rank above body-only matches.
EXAM TIP: "Search a query across the title, body, and tags fields, prioritising title" → multi_match with fields: ["title^N", "body", "tags"]
// term and terms queries
// term — exact single value (NOT analyzed)
GET /orders/_search
{
  "query": {
    "term": {
      "status": {
        "value": "pending"
      }
    }
  }
}
// Shorthand: "term": { "status": "pending" }

// terms — match any of a list of values
GET /orders/_search
{
  "query": {
    "terms": {
      "status": ["pending", "processing", "shipped"]
    }
  }
}

// terms lookup — fetch values from another doc
"terms": {
  "user_id": {
    "index": "follows",
    "id":    "user123",
    "path":  "following"
  }
}
// Notes
CRITICAL TRAP: term queries are NOT analyzed. Searching term: {"title": "Quick"} on a text field (analyzed to lowercase) will NOT match because text fields don't store the original token "Quick". Always use term on keyword fields.
keyword vs text: If you indexed "Quick Brown Fox" into a keyword field, it stores exactly "Quick Brown Fox" — a term query for "quick brown fox" (lowercase) WON'T match. Case must match exactly for keyword fields.
terms lookup: Fetches the list of values from a document in another index. Useful for "show me posts by users I follow" where the following list lives in a separate users index.
SCORING: term queries run in filter context by default inside bool/filter — no scoring overhead. In query context they contribute to relevance score.
// range query
// Numeric range
GET /products/_search
{
  "query": {
    "range": {
      "price": {
        "gte": 10,
        "lte": 100
      }
    }
  }
}

// Date range
"range": {
  "@timestamp": {
    "gte": "now-7d/d",  // 7 days ago, rounded to day
    "lt":  "now/d",     // start of today
    "time_zone": "America/New_York"
  }
}

// Date math operators:
// now-1h     = 1 hour ago
// now-7d/d   = 7 days ago, rounded down to day
// now+1M/M   = 1 month ahead, rounded to month
// 2024-01-01||+1M = specific date + 1 month
// Notes
OPERATORS: gte (≥), gt (>), lte (≤), lt (<). Mix and match: gte + lt for half-open ranges (inclusive start, exclusive end).
DATE MATH: now is always the current time in UTC. The rounding operator / rounds to the nearest unit boundary. now-7d/d rounds to the start of the day 7 days ago — consistent for caching.
FORMAT: If your date field uses a non-standard format, specify it: "format": "yyyy/MM/dd" to match the stored format.
FILTER CONTEXT: Range queries inside bool/filter don't score and are cacheable. This is the correct placement for date/numeric filters to maximise performance.
// exists and wildcard
// exists — field is present and non-null
GET /users/_search
{
  "query": {
    "exists": { "field": "email" }
  }
}

// NOT exists — use bool/must_not
"bool": {
  "must_not": [
    { "exists": { "field": "deleted_at" }}
  ]
}

// wildcard — glob patterns on keyword fields
"wildcard": {
  "email": {
    "value":         "*@elastic.co",
    "case_insensitive": true
  }
}
// ? = one char, * = zero or more chars

// prefix — keyword starts with
"prefix": {
  "username": { "value": "john" }
}

// ids — find documents by _id
"ids": { "values": ["1", "2", "3"] }
// Notes
exists vs missing: There is no "missing" query type. Use bool/must_not + exists to find documents where a field is absent or null.
WILDCARD PERFORMANCE: Leading wildcards (*@elastic.co) are very expensive — they require scanning all terms in the index. Avoid leading wildcards on large indices. prefix queries are much faster (they use the term index).
case_insensitive: true added in ES 7.10 — allows wildcard queries to be case-insensitive without a separate lowercase normalizer on the keyword field.
EXAM TIP: "Find all documents where the user has not confirmed their email" → bool/must_not + exists: {field: "confirmed_at"}
02.02Boolean Query — The Core Compound Query

The bool query is the most important compound query in Elasticsearch. Nearly every real-world search combines multiple conditions using bool. Understanding the scoring behaviour of each clause is a key exam differentiator.

// Bool Query Clause Explorer — click each clause
Scoring
must
Clause MUST match. Contributes to relevance score. Equivalent to logical AND.
No Score — Cached
filter
Clause MUST match. Does NOT score. Results are cached. Ideal for ranges, terms, dates.
Optional / Boost
should
Clause SHOULD match. Boosts score if it does. Becomes required if no must/filter clauses.
Exclude — Cached
must_not
Clause must NOT match. Does NOT score. Excludes matching documents. Cached.
// Example

          

          

            
// Key Points

            

          

        

      


      

        The canonical bool query pattern for ECE: Use filter for all non-scoring conditions (date ranges, status terms, numeric ranges). Use must only for full-text queries where relevance scoring matters. Use must_not to exclude. Use should to boost relevance without requiring a match. This pattern maximises performance through filter caching.
      


      

        

          
          
          
        

        

          

            
// Production bool query pattern

            
GET /products/_search
{
  "query": {
    "bool": {
      // MUST: scored, affects relevance
      "must": [
        { "match": { "description": "wireless headphones" }}
      ],
      // FILTER: unscored, cached, fast
      "filter": [
        { "term":  { "status": "available" }},
        { "range": { "price": { "lte": 200 }}},
        { "term":  { "category": "electronics" }}
      ],
      // SHOULD: optional, boosts relevance if match
      "should": [
        { "term": { "brand": "sony" }},
        { "term": { "features": "noise_cancelling" }}
      ],
      // MUST_NOT: excludes, unscored, cached
      "must_not": [
        { "term": { "discontinued": true }}
      ]
    }
  }
}

          

          

            
// Scoring Behaviour

            

              
SCORING RULE: Only must and should contribute to _score. filter and must_not clauses do not affect score — they only include/exclude documents.

              
should without must/filter: When a bool has ONLY should clauses (no must/filter), at least one should clause must match. minimum_should_match defaults to 1.

              
should with must/filter: When must or filter is also present, should clauses become optional — they boost score but are not required. minimum_should_match defaults to 0.

              
PERFORMANCE: filter context results are cached in the filter cache. Repeated bool queries with the same filter clause get a free cache hit. Never put frequently-repeated conditions in must — put them in filter.

            

          

        

        

          

            
// Nested bool (complex logic)

            
// Find: (shirt OR trousers) AND (red OR blue)
GET /fashion/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "bool": {
            "should": [
              { "term": { "type": "shirt"    }},
              { "term": { "type": "trousers" }}
            ]
          }
        },
        {
          "bool": {
            "should": [
              { "term": { "color": "red"  }},
              { "term": { "color": "blue" }}
            ]
          }
        }
      ]
    }
  }
}

// The inner bool/should groups act as OR blocks.
// Wrapping them in outer bool/must enforces AND.

          

          

            
// Notes

            

              
NESTING PATTERN: Bool queries can be nested to any depth. Inner bools create logical groupings. The pattern of wrapping should clauses in a must-level bool is the standard way to express (A OR B) AND (C OR D).

              
EXAM PATTERN: "Find products that are either shirts or trousers AND are either red or blue" — requires nested bool. Two inner bools, each with should (OR logic), both wrapped in an outer bool/must (AND logic).

              
CONSTANT SCORE: For must/filter combinations where you want no scoring at all, use constant_score query to wrap a filter and assign a fixed boost value.

            

          

        

        

          

            
// minimum_should_match

            
// Require at least 2 of 4 should clauses
GET /articles/_search
{
  "query": {
    "bool": {
      "should": [
        { "term": { "tags": "elasticsearch" }},
        { "term": { "tags": "search"         }},
        { "term": { "tags": "database"      }},
        { "term": { "tags": "indexing"      }}
      ],
      "minimum_should_match": 2  // 2 of 4 must match
    }
  }
}

// Other formats:
"minimum_should_match": "75%"   // percentage
"minimum_should_match": "-1"    // all - 1
"minimum_should_match": "2<50%" // conditional

// Also applies in match query:
"match": {
  "title": {
    "query":                "quick brown fox",
    "minimum_should_match": "2"
  }
}

          

          

            
// Notes

            

              
CONTROL PRECISION: minimum_should_match balances precision vs recall. Requiring more terms = higher precision, lower recall. Requiring fewer = lower precision, higher recall.

              
IN MATCH QUERY: When using operator: or (default), each term is a should clause. minimum_should_match on a match query controls how many of the analyzed terms must be found.

              
CONDITIONAL SYNTAX "2<50%" means: if 2 or fewer clauses, all must match; if more than 2, at least 50% must match. This handles variable-length input well.

            

          

        

      

    

  


  
  

    
02.03Aggregations — Metric, Bucket & Sub-Aggregations

    

      
Aggregations turn raw documents into analytics. The ECE exam tests metric aggregations (compute values: avg, min, max, sum, cardinality), bucket aggregations (group documents: terms, date_histogram, range, histogram), and sub-aggregations (nest aggs inside buckets for multi-level analytics).


      

        Key pattern: Always set "size": 0 when you only want aggregation results and not matching documents — this avoids the overhead of fetching and ranking documents, and the exam commonly asks for agg-only queries.
      


      

        
        
        
        
      


      
      

        

          
// Metric aggregations

          
GET /flights/_search
{
  "size": 0,
  "aggs": {
    "avg_price": {
      "avg": { "field": "AvgTicketPrice" }
    },
    "min_price": {
      "min": { "field": "AvgTicketPrice" }
    },
    "max_price": {
      "max": { "field": "AvgTicketPrice" }
    },
    "total_revenue": {
      "sum": { "field": "AvgTicketPrice" }
    },
    "unique_carriers": {
      "cardinality": { "field": "Carrier" }
    },
    "price_stats": {
      "stats": { "field": "AvgTicketPrice" }
      // returns: count, min, max, avg, sum
    },
    "price_percentiles": {
      "percentiles": {
        "field":    "AvgTicketPrice",
        "percents": [50, 95, 99]
      }
    }
  }
}

        

        

          
// Metric Agg Reference

          

            
avg, min, max, sum: Basic statistics on numeric fields. All work on keyword fields too if the field has doc_values enabled.

            
cardinality: Approximate count of distinct values (HyperLogLog++ algorithm). precision_threshold: 40000 for higher accuracy (uses more memory).

            
stats: Single agg returning count + min + max + avg + sum. Use extended_stats for variance and std deviation too.

            
value_count: Count of values (not distinct). top_hits: Returns top matching documents within a bucket — useful for "show me one representative document per group".

            
FIELD TYPES: Metric aggs require numeric (integer, float, etc.) or date fields. They do not work on text fields. Always aggregate on keyword or numeric.

          

        

      


      
      

        

          
// Bucket aggregations

          
// TERMS — group by distinct values
"by_carrier": {
  "terms": {
    "field": "Carrier",
    "size":  10,       // top 10
    "order": { "_count": "desc" }
  }
}

// DATE_HISTOGRAM — group by time intervals
"per_day": {
  "date_histogram": {
    "field":             "timestamp",
    "calendar_interval": "1d",  // day/week/month/year
    "min_doc_count":     0,    // include empty buckets
    "time_zone":         "Europe/London"
  }
}

// RANGE — custom numeric buckets
"price_ranges": {
  "range": {
    "field": "price",
    "ranges": [
      { "to": 50                    },
      { "from": 50,  "to": 200    },
      { "from": 200                  }
    ]
  }
}

// HISTOGRAM — fixed-width numeric buckets
"price_hist": {
  "histogram": {
    "field":    "price",
    "interval": 50
  }
}

        

        

          
// Bucket Agg Notes

          

            
terms agg: Groups by top N distinct values. size controls how many buckets are returned. Default order is by doc count descending. You can order by a sub-aggregation metric: "order": {"avg_price": "desc"}.

            
terms accuracy: Terms agg is an approximation for distributed indices. Increase shard_size (default: size * 1.5 + 10) for higher accuracy at cost of more data transferred from shards.

            
calendar_interval vs fixed_interval: Use calendar_interval for human-readable buckets (1d, 1w, 1M, 1q, 1y). Use fixed_interval for precise durations (60m, 24h) that don't align to calendar boundaries.

            
missing: All bucket aggs support a missing parameter — documents where the field doesn't exist go into a "missing" bucket: "missing": "N/A".

          

        

      


      
      

        

          
// Sub-aggregations

          
// Average price per carrier, sorted by avg
GET /flights/_search
{
  "size": 0,
  "aggs": {
    "by_carrier": {
      "terms": {
        "field": "Carrier",
        "size":  10,
        "order": { "avg_price": "desc" }
      },
      "aggs": {               // sub-agg runs per bucket
        "avg_price": {
          "avg": { "field": "AvgTicketPrice" }
        },
        "dest_breakdown": {
          "terms": {           // nested bucket agg!
            "field": "DestCountry",
            "size":  5
          },
          "aggs": {
            "flight_count": {  // 3 levels deep
              "value_count": { "field": "FlightNum" }
            }
          }
        }
      }
    }
  }
}

        

        

          
// Sub-Agg Notes

          

            
NESTING: Sub-aggs are defined inside a parent bucket agg using the aggs key at the same level as the bucket type. They execute once per bucket produced by the parent.

            
ORDER BY METRIC: You can sort a terms bucket by a sub-agg metric value. "order": {"avg_price": "desc"} sorts buckets by the value of the avg_price sub-agg rather than by doc count.

            
SCOPE: Each sub-agg only sees documents in its parent bucket. A date_histogram bucket for "January" only aggregates January documents in its sub-aggs.

            
PERFORMANCE: Deep nesting (3+ levels) on high-cardinality fields can be expensive. Limit terms agg sizes at each level. Consider whether the full nesting is truly needed.

          

        

      


      
      

        

          
// Pipeline aggregations

          
// avg_bucket — average across buckets
GET /flights/_search
{
  "size": 0,
  "aggs": {
    "monthly": {
      "date_histogram": {
        "field":             "timestamp",
        "calendar_interval": "month"
      },
      "aggs": {
        "total_rev": {
          "sum": { "field": "AvgTicketPrice" }
        }
      }
    },
    // Pipeline agg — references sibling bucket
    "avg_monthly_rev": {
      "avg_bucket": {
        "buckets_path": "monthly>total_rev"
      }
    },
    "max_monthly_rev": {
      "max_bucket": {
        "buckets_path": "monthly>total_rev"
      }
    },
    "cumulative_rev": {
      "cumulative_sum": {
        "buckets_path": "monthly>total_rev"
      }
    }
  }
}

        

        

          
// Pipeline Agg Notes

          

            
PIPELINE AGGS: Operate on the output of other aggregations (not on documents directly). They take a buckets_path pointing to a sibling or parent agg's output.

            
buckets_path syntax: Use > to navigate levels. "monthly>total_rev" = the total_rev agg inside the monthly bucket. Parent pipeline aggs use "_count" or "_key" to reference bucket metadata.

            
COMMON PIPELINE AGGS: avg_bucket, max_bucket, min_bucket, sum_bucket (across buckets), cumulative_sum (running total), derivative (rate of change between buckets), moving_avg (smoothing).

            
SCOPE: Pipeline aggs are at the same level as the bucket they reference — they are siblings, not sub-aggs. They cannot go inside a bucket; they reference sibling bucket output.

          

        

      

    

  


  
  

    
02.04Async Search · Cross-Cluster · Runtime Fields

    

      

        

          
          
          
        


        
        

          

            
// Asynchronous search

            
// Submit async search
POST /large_index/_async_search
{
  "wait_for_completion_timeout": "1s",
  "keep_on_completion": true,
  "keep_alive":          "5d",
  "query": {
    "match": { "message": "error" }
  },
  "aggs": {
    "daily_errors": {
      "date_histogram": {
        "field":             "@timestamp",
        "calendar_interval": "day"
      }
    }
  }
}
// Returns immediately with id + partial results
// { "id": "FmRldE8...", "is_running": true, ... }

// Poll for results
GET /_async_search/FmRldE8...

// Delete when done (free resources)
DELETE /_async_search/FmRldE8...

// Check status without fetching results
GET /_async_search/status/FmRldE8...

          

          

            
// Notes

            

              
USE CASE: Long-running aggregation or search on large indices. Submit with wait_for_completion_timeout: 0 to return immediately, then poll until is_running: false.

              
PARTIAL RESULTS: While running, GET on the async ID returns partial results as they accumulate. response.is_partial: true means data is incomplete. is_running: false + is_partial: false = complete results.

              
keep_alive: How long to retain results after completion (default 5d). Set keep_on_completion: false to auto-delete after retrieval.

              
EXAM TASK: "Create and execute an asynchronous search with wait_for_completion_timeout: 0, then fetch the results using the returned ID" — two separate API calls: POST _async_search, then GET /_async_search/{id}.

            

          

        


        
        

          

            
// Cross-cluster search (CCS)

            
// Prerequisites: configure remote cluster
PUT /_cluster/settings
{
  "persistent": {
    "cluster": {
      "remote": {
        "cluster_two": {
          "seeds": ["10.0.1.105:9300"],
          "skip_unavailable": true
        }
      }
    }
  }
}

// Search across clusters: cluster:index syntax
GET /local_index,cluster_two:remote_index/_search
{
  "query": {
    "match": { "message": "error" }
  }
}

// Search all indices on both clusters
GET /logs-*,cluster_two:logs-*/_search
{ "query": { "match_all": {} } }

// Wildcard: all remote clusters
GET /*:logs-*/_search
{ "query": { "match_all": {} } }

          

          

            
// Notes

            

              
SYNTAX: remote_cluster_name:index_name — the colon separates cluster from index. Multiple indices from multiple clusters comma-separated in the URL.

              
skip_unavailable: true — if the remote cluster is down, the search still runs against local indices (returns partial results). With false (default), search fails if remote is unavailable.

              
TRANSPORT PORT: CCS uses the transport port (9300 default), not the HTTP port (9200). Remote seeds must use host:9300 format.

              
VERIFY: GET /_remote/info — shows all configured remote clusters and their connection status. Always verify after configuring before attempting a CCS query.

            

          

        


        
        

          

            
// Runtime fields in search

            
// Define runtime field and use in query + aggs
GET /logs/_search
{
  "runtime_mappings": {
    "is_error": {
      "type": "boolean",
      "script": """
        emit(doc['status_code'].value >= 500)
      """
    },
    "duration_seconds": {
      "type": "double",
      "script": """
        emit(doc['duration_ms'].value / 1000.0)
      """
    }
  },
  "query": {
    "bool": {
      "filter": [
        { "term": { "is_error": true }}
      ]
    }
  },
  "aggs": {
    "avg_duration_s": {
      "avg": { "field": "duration_seconds" }
    }
  },
  "fields": ["is_error", "duration_seconds"]
}

          

          

            
// Notes

            

              
FULL INTEGRATION: Runtime fields defined in runtime_mappings can be used in queries, filters, aggregations, sorting, and returned in the fields array — all in the same request.

              
emit(): The Painless emit() function outputs the computed value for the runtime field. For boolean: emit(condition). For double: emit(numericValue). Must use emit, not return.

              
FIELDS vs _source: Runtime field values appear in the fields array of the response, not in _source (since they're computed, not stored). Access them via response.hits.hits[].fields.

              
PERFORMANCE: Runtime fields execute a Painless script on every matching document. For large result sets, this is expensive. Index the field properly if the computation is needed on every query.

Bool Query · Scoring

              
Search the ecommerce index for products in the "clothing" category priced between $20–$100, that are in stock. Boost results where the product name contains "premium". Return only 10 results, showing only the product name and price fields.

              
Hint: mix filter (no score) for category/price/stock, must for text match, should for boost

            

          

          

            

              
// Solution

              
GET /ecommerce/_search
{
  "size": 10,
  "_source": ["product_name", "price"],
  "query": {
    "bool": {
      "filter": [
        { "term":  { "category":  "clothing" }},
        { "term":  { "in_stock":  true }},
        { "range": { "price": { "gte": 20, "lte": 100 }}}
      ],
      "should": [
        { "match": { "product_name": "premium" }}
      ]
    }
  }
}

              
Why filter not must: category, in_stock, and price range are hard constraints — documents either match or they don't. Using filter means no scoring overhead and results are cached. The "premium" should clause boosts score for matching docs but doesn't exclude non-matching ones. _source with an array limits which fields are returned in the response.

            

          

        


        

          

            
02

            

              
Aggregations · Sub-Agg · Sort by Metric

              
On the flights index, find the top 5 destination countries by average ticket price (descending). For each country, also show the total number of flights and the maximum ticket price.

              
Hint: terms agg ordered by a sub-agg metric, with multiple metric sub-aggs

            

          

          

            

              
// Solution

              
GET /flights/_search
{
  "size": 0,
  "aggs": {
    "top_dest_countries": {
      "terms": {
        "field": "DestCountry",
        "size":  5,
        "order": { "avg_price": "desc" }
      },
      "aggs": {
        "avg_price": {
          "avg": { "field": "AvgTicketPrice" }
        },
        "total_flights": {
          "value_count": { "field": "FlightNum" }
        },
        "max_price": {
          "max": { "field": "AvgTicketPrice" }
        }
      }
    }
  }
}

              
Ordering by sub-agg: The terms agg's "order" key references the sub-agg name "avg_price" (defined inside aggs). size: 0 skips document hits. value_count counts non-null values of a field per bucket — an efficient flight counter. All three sub-aggs run in parallel for each of the top 5 country buckets.

            

          

        


        

          

            
03

            

              
Async Search · Cross-Cluster

              
Configure cluster c2 as a remote cluster (seeds: 10.0.1.105:9300). Then run an asynchronous search against the filebeat-* index on BOTH the local cluster and c2 simultaneously, searching for log messages mentioning "disk full". Set wait_for_completion_timeout to 0. Fetch the results using the returned ID.

              
Hint: cluster settings, then CCS with async search together

            

          

          

            

              
// Solution

              
// Step 1: Configure remote cluster
PUT /_cluster/settings
{
  "persistent": {
    "cluster.remote.c2.seeds": ["10.0.1.105:9300"]
  }
}

// Step 2: Verify remote cluster connection
GET /_remote/info

// Step 3: Async cross-cluster search
POST /filebeat-*,c2:filebeat-*/_async_search
{
  "wait_for_completion_timeout": "0",
  "keep_on_completion": true,
  "query": {
    "match": { "message": "disk full" }
  }
}
// Returns: { "id": "FmRldE8...", "is_running": true }

// Step 4: Fetch results by ID
GET /_async_search/FmRldE8...

              
Key points: CCS syntax is cluster_name:index_pattern — local indices come first without prefix, remote indices use cluster_name: prefix. Async search and CCS can be combined in one request. wait_for_completion_timeout: "0" returns immediately. Always call GET /_remote/info to confirm connectivity before running CCS in the exam.

            

          

        


        

          

            
04

            

              
Runtime Field · Aggregation

              
The logs index has a response_time_ms field (integer). Using a runtime field, compute response_time_s (as a double) and aggregate: show average, max, and 95th percentile response time in seconds — all without modifying the index mapping.

              
Hint: runtime_mappings in the search body + metric aggs referencing the runtime field

            

          

          

            

              
// Solution

              
GET /logs/_search
{
  "size": 0,
  "runtime_mappings": {
    "response_time_s": {
      "type": "double",
      "script": {
        "source": "emit(doc['response_time_ms'].value / 1000.0)"
      }
    }
  },
  "aggs": {
    "avg_response_s": {
      "avg": { "field": "response_time_s" }
    },
    "max_response_s": {
      "max": { "field": "response_time_s" }
    },
    "p95_response_s": {
      "percentiles": {
        "field":    "response_time_s",
        "percents": [95]
      }
    }
  }
}

              
Why this works without schema changes: runtime_mappings in the request body creates an ephemeral field for the duration of this search only. The Painless script divides ms by 1000.0 (float division) to get seconds as a double. All three aggregations reference the runtime field name "response_time_s" just like any indexed field.

8 Exam-Style Questions · Searching Data Domain

      

    


    
    

      
QUESTION 01

      
A status field is mapped as keyword and contains the value "Active". Which query will correctly match documents with this value?

      

        
        
        
        
      

      
Correct: term query with exact case "Active". Keyword fields are not analyzed — they store the exact value as indexed. A term query does not analyze the search value, so term: "Active" matches the stored "Active" exactly. A match query analyzes the search string (lowercases it to "active"), which won't match "Active" in a keyword field. Option D fails because term queries are also not analyzed — "active" (lowercase) won't match "Active" stored in a keyword field.

    


    
    

      
QUESTION 02

      
You want to search for products where the description contains "wireless headphones" AND the price is between $50–$300 AND the brand is "Sony". Which bool clause placement gives the best query performance?

      

        
        
        
        
      

      
Correct: description in must, price and brand in filter. The full-text query on description belongs in must — it's a text match where relevance scoring matters. The price range and brand term are exact/structured queries where you only care about inclusion/exclusion, not scoring. Placing them in filter means: (1) no scoring overhead for those conditions, (2) results are cached in the filter cache for reuse. Putting everything in must scores every condition, which is unnecessary and slower for structured filters.

    


    
    

      
QUESTION 03

      
You run the following aggregation. What does the order key accomplish?

      
"top_products": {
  "terms": {
    "field": "category",
    "size": 5,
    "order": { "avg_revenue": "desc" }
  },
  "aggs": {
    "avg_revenue": {
      "avg": { "field": "revenue" }
    }
  }
}

      

        
        
        
        
      

      
Correct: returns the top 5 categories ranked by average revenue. When a terms agg has order pointing to a sub-agg name, Elasticsearch uses that sub-agg's value to determine which buckets are "top" — instead of the default doc count ordering. The result is the 5 categories with the highest average revenue, not the 5 with the most documents. Option B is wrong because the ordering happens at selection time, not after.

    


    
    

      
QUESTION 04

      
Which of the following correctly represents the cross-cluster search syntax to query the logs-2024 index on a remote cluster named dc2?

      

        
        
        
        
      

      
Correct: dc2:logs-2024. Cross-cluster search uses the cluster_name:index_name colon-separated syntax directly in the URL path. The cluster name must match a configured remote cluster name (from PUT /_cluster/settings persistent.cluster.remote). There are no special headers or body parameters for CCS — the cluster:index syntax in the URL is sufficient.

    


    
    

      
QUESTION 05

      
What is the difference between should in a bool query that also has a must clause, versus a bool query with ONLY should clauses?

      

        
        
        
        
      

      
Correct: D. When a bool query has no must or filter clauses and only should clauses, minimum_should_match defaults to 1 — at least one should must match for a document to be included. When a must or filter is also present, should clauses become purely optional (minimum_should_match defaults to 0) — they only influence the relevance score of documents that are already required to match by the must/filter. This is a critical exam distinction.

    


    
    

      
QUESTION 06

      
An asynchronous search returns "is_running": false, "is_partial": false. What does this indicate?

      

        
        
        
        
      

      
Correct: search completed, results are complete. For async searches: is_running: true = still executing. is_running: false, is_partial: true = finished but some shards failed or timed out (partial results). is_running: false, is_partial: false = fully complete, all shards responded, full results available. This is the final state you should check for before consuming the results.

    


    
    

      
QUESTION 07

      
What is the correct way to find all documents where a field called archived_at does NOT exist (is absent or null)?

      

        
        
        
        
      

      
Correct: bool/must_not + exists. There is no "missing" query in Elasticsearch (it was removed). The correct idiom for "field does not exist or is null" is to negate the exists query using bool/must_not. This is a very common pattern in the exam. Option B is invalid syntax. Option D would search for the string literal "null" in a field, not for absent fields.

    


    
    

      
QUESTION 08

      
You want to aggregate daily flight counts over the last 30 days, including days with zero flights (no documents). Which date_histogram parameter ensures empty days appear in results?

      

        
        
        
        
      

      
Correct: min_doc_count: 0 with extended_bounds. By default, date_histogram only returns buckets that contain at least one document. Setting min_doc_count: 0 includes empty buckets, but only between the observed min and max dates in the data. To also include buckets at the edges of your desired range (e.g., days before the first or after the last document), you must also set extended_bounds with explicit min/max values. "include_empty_buckets" is not a real parameter.
